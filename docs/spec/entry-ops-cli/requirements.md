# entry-ops-cli 要件定義書

## 概要

Sprint 4「エントリ操作 + CLI」の要件定義書。vault.pqd内のエントリに対するCRUD操作、パスワード入力の3段階方式、$EDITORによる安全な編集、ネストタグ機能を定義する。

## 機能要件（EARS記法）

**【信頼性レベル凡例】**:
- 🔵 **青信号**: PRD・設計文書・ユーザヒアリングを参考にした確実な要件
- 🟡 **黄信号**: PRD・設計文書・ユーザヒアリングから妥当な推測による要件
- 🔴 **赤信号**: PRD・設計文書・ユーザヒアリングにない推測による要件

### 通常要件 — エントリCRUD

- **REQ-001**: `new` コマンド実行時、システムはタイトル・本文・タグを受け取り、UUID v4を生成し、本文をAES-256-GCMで暗号化し、ML-DSA-65で署名し、HMAC-SHA256でcontent_hmacを計算し、EntryRecordとしてvault.pqdに追記しなければならない 🔵 *PRD §2.3, §4.1, §6.2より*

- **REQ-002**: `list` コマンド実行時、システムはvault.pqd内の全エントリを復号し、デフォルトで新しい順に20件を表示しなければならない。各行にはIDプレフィックス（4〜8文字）・日付・タイトル・タグを含む 🔵 *PRD §4.1 + ヒアリングQ4より*

- **REQ-003**: `show <ID_PREFIX>` 実行時、システムは最小4文字のプレフィックスでエントリを一意に特定し、復号して本文全体を表示しなければならない 🔵 *PRD §4.1 + ヒアリングQ3より*

- **REQ-004**: `edit <ID_PREFIX>` 実行時、システムはエントリを復号し、ヘッダーコメント形式（`# Title:` / `# Tags:` / `# ---` + 本文）で安全な一時ファイルに書き出し、$EDITORを起動しなければならない。編集後、ヘッダーをパースしてメタデータを更新し、本文を再暗号化・再署名してvault.pqdを更新しなければならない 🔵 *PRD §4.1, §4.3 + ヒアリングQ8より*

- **REQ-005**: `delete <ID_PREFIX>` 実行時、システムはエントリのタイトル・作成日を表示して確認プロンプトを出し、ユーザーが承認した場合にのみvault.pqdからエントリを削除しなければならない 🔵 *PRD §4.1 + ヒアリングQ2より*

### 条件付き要件 — new入力優先順位

- **REQ-011**: `-b BODY` フラグが指定された場合、システムはその値を本文として使用し、$EDITORを起動してはならない 🔵 *ヒアリングQ7より*

- **REQ-012**: `-b` フラグが未指定かつstdinがパイプ入力の場合、システムはstdinから本文を読み取り、$EDITORを起動してはならない 🔵 *ヒアリングQ7より*

- **REQ-013**: `-b` フラグが未指定かつstdinがTTYの場合、システムは$EDITORを起動して本文を入力させなければならない 🔵 *ヒアリングQ7より*

### 条件付き要件 — edit CLIフラグ

- **REQ-014**: `edit <ID_PREFIX> --title "新タイトル"` が指定された場合、システムは$EDITORを起動せずにタイトルのみを変更しなければならない 🔵 *ヒアリングQ8より*

- **REQ-015**: `edit <ID_PREFIX> --add-tag "タグ"` が指定された場合、システムは$EDITORを起動せずにタグを追加しなければならない 🔵 *ヒアリングQ8より*

- **REQ-016**: `edit <ID_PREFIX> --remove-tag "タグ"` が指定された場合、システムは$EDITORを起動せずにタグを削除しなければならない 🔵 *ヒアリングQ8より*

### 条件付き要件 — delete確認

- **REQ-021**: `--force` / `-f` フラグが指定された場合、システムは確認プロンプトを表示せずに即座に削除しなければならない 🔵 *ヒアリングQ2より*

- **REQ-022**: `--claude` フラグが有効な場合、deleteは `--force` 相当として確認プロンプトをスキップしなければならない 🔵 *ヒアリングQ2 + PRD §4.4より*

### 条件付き要件 — list フィルタ

- **REQ-031**: `--tag TAG` が指定された場合、システムは指定タグに前方一致するエントリのみを表示しなければならない 🔵 *PRD §4.1, backlog S4より*

- **REQ-032**: `-q QUERY` が指定された場合、システムはタイトルにクエリ文字列を含むエントリのみを表示しなければならない（本文検索はS6のsearchコマンドで実装） 🔵 *PRD §4.1 + ヒアリング確認済み*

- **REQ-033**: `-n N` が指定された場合、システムは表示件数をN件に制限しなければならない 🔵 *PRD §4.1 + ヒアリングQ4より*

### 条件付き要件 — IDプレフィックス解決

- **REQ-041**: ID_PREFIXが4文字未満の場合、システムはエラー「IDプレフィックスは4文字以上必要です」を返さなければならない 🔵 *ヒアリングQ3より*

- **REQ-042**: ID_PREFIXが複数のエントリにマッチする場合、システムはエラーとともにマッチした候補一覧を表示しなければならない 🔵 *ヒアリングQ3より*

- **REQ-043**: ID_PREFIXがどのエントリにもマッチしない場合、システムはエラー「エントリが見つかりません」を返さなければならない 🔵 *PRD §4.1から妥当な推測*

### 通常要件 — パスワード入力

- **REQ-051**: システムは以下の優先順位でパスワードを取得しなければならない: (1) `--password` フラグ (2) `PQ_DIARY_PASSWORD` 環境変数 (3) TTYプロンプト 🔵 *PRD §4.2, ADR-0003より*

- **REQ-052**: `--password` フラグ使用時、システムは「シェル履歴に平文が記録されるリスクがあります」と警告を表示しなければならない 🔵 *PRD §4.2より*

- **REQ-053**: TTYプロンプトでのパスワード入力は、入力文字をエコーせず（Unix: termios ECHO無効化 / Windows: SetConsoleMode ENABLE_ECHO_INPUT無効化）、入力を直接SecretStringに格納しなければならない 🔵 *ADR-0003 + ヒアリングQ6より*

- **REQ-054**: stdinがTTYでなく、`--password` もenv変数も未設定の場合、システムはエラー「パスワードが指定されていません。--password, PQ_DIARY_PASSWORD環境変数, またはTTYで入力してください」を返さなければならない 🔵 *Bitwarden CLI参考 + ヒアリング確認済み*

### 通常要件 — $EDITOR一時ファイル制御

- **REQ-061**: システムはセキュアな一時ディレクトリに一時ファイルを作成しなければならない。Unix: `/dev/shm` > `/run/user/$UID/` > `/tmp`（フォールバック時は警告）。Windows: `%LOCALAPPDATA%\pq-diary\tmp\` 🔵 *PRD §4.3 + ヒアリングQ5より*

- **REQ-062**: vim/neovim使用時、システムは `-c 'set noswapfile nobackup noundofile'` オプションを渡さなければならない 🔵 *PRD §4.3より*

- **REQ-063**: $EDITOR起動時、システムは `$TMPDIR` / `$TEMP` / `$TMP` 環境変数をセキュア一時ディレクトリに上書きしなければならない 🔵 *PRD §4.3より*

- **REQ-064**: 編集完了後（正常・異常問わず）、システムは一時ファイルをランダムデータで上書きしてからファイルを削除しなければならない 🔵 *PRD §4.3, §16.1 攻撃Gより*

- **REQ-065**: Windows環境では、セキュア一時ディレクトリに対してACLでオーナーのみ読み書き権限を設定しなければならない 🔵 *ヒアリングQ5より*

### 通常要件 — ネストタグ

- **REQ-071**: システムは `#親/子/孫` 形式のネストタグをサポートしなければならない 🔵 *PRD backlog S4, Obsidian互換*

- **REQ-072**: タグ名には英数字・Unicode文字・`_`・`-`・`/`（階層区切り）を許容し、スペースは禁止しなければならない 🔵 *Obsidianタグ命名規則準拠*

- **REQ-073**: `--tag` フィルタは前方一致で動作しなければならない。`--tag "日記"` は `#日記`, `#日記/振り返り`, `#日記/仕事` すべてにマッチする 🔵 *PRD backlog S4「前方一致検索」*

### 通常要件 — データモデル

- **REQ-081**: エントリの暗号化ペイロードはメタデータ分離型とする。暗号化対象は `{ title: String, tags: Vec<String>, body: String }` を含む構造体をシリアライズしたバイト列とする 🔵 *ヒアリングQ1より*

- **REQ-082**: $EDITORでの編集時、一時ファイルはヘッダーコメント形式で出力しなければならない: `# Title: {title}\n# Tags: {tag1}, {tag2}\n# ---\n\n{body}` 🔵 *ヒアリングQ8より*

- **REQ-083**: 編集後の一時ファイルからヘッダーをパースする際、ヘッダーが不正な場合はエラーとし、元のメタデータを保持して本文の変更のみを適用しなければならない 🔵 *ヒアリング確認済み*

### 通常要件 — Vault操作

- **REQ-091**: `-v VAULT` グローバルオプションが指定されない場合、config.tomlの `[defaults].vault` を使用しなければならない 🔵 *PRD §3.3, §4.1より*

- **REQ-092**: エントリ操作前にパスワードでVaultを解錠し、操作完了後に鍵をzeroizeしなければならない 🔵 *PRD §1.2 設計原則2, §8.2より*

## 非機能要件

### パフォーマンス

- **NFR-001**: `new` / `edit` は200ms以内に完了しなければならない（$EDITOR起動時間を除く） 🔵 *PRD §12.1より*
- **NFR-002**: `list` は100エントリで500ms以内に完了しなければならない 🔵 *PRD §12.1より*

### セキュリティ

- **NFR-101**: パスワード・復号済み本文・対称鍵は操作完了後に即座にzeroizeしなければならない 🔵 *PRD §1.2 設計原則2, §8.1より*
- **NFR-102**: `--password` フラグの値は使用後即座にzeroizeしなければならない 🔵 *PRD §4.2より*
- **NFR-103**: 一時ファイルの内容は操作完了後にランダムデータで上書き削除しなければならない 🔵 *PRD §4.3より*

### クロスプラットフォーム

- **NFR-201**: TTYパスワード入力はUnix (termios) / Windows (SetConsoleMode) の両方で動作しなければならない 🔵 *ADR-0003 + ヒアリングQ6より*
- **NFR-202**: セキュア一時ディレクトリはUnix (/dev/shm等) / Windows (%LOCALAPPDATA%) の両方で動作しなければならない 🔵 *ヒアリングQ5より*

## Edgeケース

### エラー処理

- **EDGE-001**: $EDITORが未設定かつTTYの場合、`vi` をフォールバックとして使用する（Unix）。Windowsでは `notepad` をフォールバックとする 🔵 *ヒアリング確認済み*
- **EDGE-002**: $EDITORが異常終了（非0）した場合、編集を破棄してエラーを返す。一時ファイルはzeroize削除する 🔵 *ヒアリング確認済み*
- **EDGE-003**: $EDITORで内容が変更されなかった場合（保存せずに終了）、エントリを更新せずに終了する 🔵 *ヒアリング確認済み（git commitと同じ動作）*

### 境界値

- **EDGE-101**: タイトルが空文字列の場合、「Untitled」をデフォルトタイトルとする 🔵 *ヒアリング確認済み*
- **EDGE-102**: タグが空リストの場合、タグなしエントリとして有効に作成する 🔵 *PRD §4.1 `-t TAG` がオプションであることから*
- **EDGE-103**: 本文が空の場合、空のエントリとして有効に作成する 🔵 *ヒアリング確認済み*
- **EDGE-104**: vault.pqd内にエントリが0件の場合、`list` は空リストを表示しエラーとしない 🔵 *一般的なCLIツールの動作*

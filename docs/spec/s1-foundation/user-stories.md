## エピック1: プロジェクト基盤

### ストーリー 1.1: ワークスペース構成 🔵

**信頼性**: 🔵 *PRDセクション2.1より*

**私は** 開発者 **として**
**core/ と cli/ が分離されたCargoワークスペースを構築したい**
**そうすることで** ドメインロジックとプラットフォーム依存コードが明確に分離され、将来のUniFFI対応が容易になる

**関連要件**: REQ-001, REQ-401

**詳細シナリオ**:
1. `Cargo.toml` でワークスペースメンバーとして `core/` と `cli/` を定義
2. `core/Cargo.toml` で `pq-diary-core` ライブラリクレートを定義
3. `cli/Cargo.toml` で `pq-diary` バイナリクレートを定義し、`pq-diary-core` に依存
4. `cargo build --workspace` が成功する

**優先度**: Must Have

---

### ストーリー 1.2: エラー型定義 🔵

**信頼性**: 🔵 *PRD + ヒアリングQ3より*

**私は** 開発者 **として**
**全Phase分のエラーバリアントを持つ統一エラー型を定義したい**
**そうすることで** 後続Sprintでエラー型を追加する必要がなく、一貫したエラーハンドリングができる

**関連要件**: REQ-002, REQ-403

**詳細シナリオ**:
1. `core/src/error.rs` に `DiaryError` 列挙型を定義
2. 各バリアントに `#[error("...")]` で英語のエラーメッセージを付与
3. 外部エラー型 (`std::io::Error`, `toml::de::Error` 等) からの `#[from]` 変換を定義
4. cli/ では `anyhow::Result` でラップして使用

**優先度**: Must Have

---

### ストーリー 1.3: セキュアメモリ型 🔵

**信頼性**: 🔵 *PRDセクション8.1 + ヒアリングQ1より*

**私は** 開発者 **として**
**Drop時に自動ゼロ埋めされるセキュアメモリ型を実装したい**
**そうすることで** 秘密データがメモリに残存するリスクを最小化できる

**関連要件**: REQ-003, REQ-402

**詳細シナリオ**:
1. `SecureBuffer` を実装: `Vec<u8>` をラップし `ZeroizeOnDrop` を派生
2. `ZeroizingKey` を実装: `[u8; 32]` をラップし `ZeroizeOnDrop` を派生
3. `MasterKey` 構造体を定義: `sym_key`, `dsa_sk`, `kem_sk` フィールド
4. `CryptoEngine` 構造体を定義: `Option<Secret<MasterKey>>` で鍵を保持
5. テストで Drop 後のzeroize動作を検証

**前提条件**:
- `zeroize` クレートと `secrecy` クレートが依存に追加されていること

**優先度**: Must Have

## エピック2: CLIインターフェース

### ストーリー 2.1: 全コマンドスケルトン 🔵

**信頼性**: 🔵 *PRDセクション4.1 + ヒアリングQ2より*

**私は** ユーザー **として**
**`pq-diary --help` で全コマンドの一覧を確認したい**
**そうすることで** ツールの全機能を把握でき、将来のコマンドも予め見える

**関連要件**: REQ-004

**詳細シナリオ**:
1. `pq-diary --help` で全サブコマンドが表示される
2. `pq-diary vault --help` でvault関連サブコマンドが表示される
3. `pq-diary new --help` でオプション (`-t`, `-b`, `-v` 等) が表示される
4. 未実装コマンド実行時に `"Planned for Sprint N"` メッセージが表示される

**優先度**: Must Have

## エピック3: 品質基盤

### ストーリー 3.1: CIパイプライン 🔵

**信頼性**: 🔵 *ヒアリングQ4より*

**私は** 開発者 **として**
**PRごとに自動でビルド・テスト・lint・脆弱性チェックが走るCIが欲しい**
**そうすることで** 品質低下を早期に検知でき、DoDの自動検証ができる

**関連要件**: REQ-005, REQ-101, REQ-102

**詳細シナリオ**:
1. `push` / `pull_request` トリガーでCIが起動
2. ubuntu-latest で `cargo build`, `cargo test`, `cargo clippy`, `cargo audit` を実行
3. いずれかが失敗した場合、PRマージをブロック

**優先度**: Must Have

## エピック 1: 初回セットアップ体験

### ストーリー 1.1: 新規ユーザーの初回起動 🔵

**信頼性**: 🔵 *PRD L218 + 2026-05-17 ヒアリング*

**私は** pq-diary を初めて使うユーザー **として**
**`pq-diary init` 一発でセットアップを完了したい**
**そうすることで** 複雑な vault create 引数を覚えずに最初のエントリを書き始められる

**関連要件**: REQ-101, REQ-102, REQ-103, REQ-104, NFR-001, NFR-201

**詳細シナリオ**:
1. ユーザーが端末で `pq-diary init` を実行
2. システムが新パスワードを TTY で 2 回求める
3. システムが `~/.pq-diary/config.toml` と `~/.pq-diary/vaults/default/` を作成
4. システムが `Initialized pq-diary at ~/.pq-diary/` と表示
5. ユーザーが続けて `pq-diary new "最初のエントリ"` を実行できる (vault 指定不要、AppConfig の default_vault が使われる)

**前提条件**:
- `~/.pq-diary/` がまだ存在しない、または `config.toml` が無い状態
- TTY が利用可能 (パスワード入力のため)

**制約事項**:
- init は 3 秒以内に完了 (NFR-001)
- パスワード入力は履歴に残らない

**優先度**: Must Have

### ストーリー 1.2: 二度目以降の init 試行を防ぐ 🔵

**信頼性**: 🔵 *2026-05-17 ヒアリング*

**私は** すでに pq-diary を設定済みのユーザー **として**
**誤って `pq-diary init` を再実行しても既存データが上書きされないでほしい**
**そうすることで** 既存の vault と全エントリを失わずに済む

**関連要件**: REQ-111

**詳細シナリオ**:
1. すでに `~/.pq-diary/config.toml` が存在する状態でユーザーが `pq-diary init` を実行
2. システムが `Error: Already initialized at ~/.pq-diary/` と表示して非ゼロ終了
3. 既存の config.toml / vault / エントリは一切変更されない

**前提条件**:
- `~/.pq-diary/config.toml` が既に存在

**優先度**: Must Have

## エピック 2: 同期の高レベル抽象化

### ストーリー 2.1: バックエンドを意識しない同期 🔵

**信頼性**: 🔵 *PRD L230 + 2026-05-17 ヒアリング*

**私は** 複数バックエンド (Git, 将来は Unix ソケット等) を切り替えながら使うユーザー **として**
**`pq-diary sync` 一発で設定されたバックエンドへ同期したい**
**そうすることで** `git-sync` / `daemon sync` を毎回打ち分けずに済む

**関連要件**: REQ-201, REQ-202, REQ-212

**詳細シナリオ**:
1. ユーザーが `pq-diary sync` を実行
2. システムが `~/.pq-diary/config.toml` から `sync_backend` を読む
3. `sync_backend = "git"` (またはデフォルト) なら `git_sync` を呼ぶ
4. Pull → Push のシーケンスが実行される

**前提条件**:
- `pq-diary init` 済み
- vault が `pq-diary git-init` で git 連携セットアップ済み

**優先度**: Must Have

## エピック 3: マスターパスワード変更

### ストーリー 3.1: パスワード漏洩疑いへの対処 🔵

**信頼性**: 🔵 *PRD L234*

**私は** マスターパスワードが漏洩した可能性があるユーザー **として**
**`pq-diary change-password` で全エントリを再暗号化したい**
**そうすることで** 旧パスワードでは復号できない状態に移行できる

**関連要件**: REQ-301, REQ-302, REQ-303, REQ-304, REQ-305, NFR-003, NFR-101, NFR-105

**詳細シナリオ**:
1. ユーザーが `pq-diary change-password` を実行
2. システムが旧パスワードを TTY で取得
3. 旧パスワードで vault.pqd を復号 (失敗時 REQ-311)
4. システムが新パスワードを 2 回 TTY で取得 (確認入力)
5. システムが全エントリと vault ヘッダーを新パスワードで再暗号化し `vault.pqd.tmp` に書き出し
6. `rename(vault.pqd.tmp, vault.pqd)` で原子的に差し替え
7. システムが `Password changed successfully` と表示

**前提条件**:
- vault が存在
- ディスクに vault.pqd の 2 倍のサイズの空きがある (tmp + 既存)

**制約事項**:
- 中断 (SIGINT, クラッシュ) しても旧 vault.pqd は無傷で残る
- 旧鍵・新鍵・全エントリは ZeroizingKey/SecretBytes でラップ

**優先度**: Must Have

---

### ストーリー 3.2: 中断時の安全な復旧 🔵

**信頼性**: 🔵 *2026-05-17 ヒアリング*

**私は** PC が突然落ちる可能性のあるユーザー **として**
**`change-password` の途中で電源断しても vault が破損しないでほしい**
**そうすることで** バックアップ無しでも次回起動で旧パスワードで開ける

**関連要件**: REQ-313, EDGE-003

**詳細シナリオ**:
1. ユーザーが `change-password` を実行し、再暗号化中に PC が停止
2. ディスク上には旧 `vault.pqd` と書きかけの `vault.pqd.tmp` が残る
3. 次回起動時に旧パスワードで vault.pqd が開ける (一切変更されていない)
4. ユーザーは手動で `vault.pqd.tmp` を削除するか、再度 `change-password` を試行する

**前提条件**:
- ファイルシステムが POSIX rename のアトミック性を保証している

**優先度**: Must Have

**信頼性**: 🔵 *ヒアリング2026-05-17 確定提案E*

**関連要件**: REQ-314, EDGE-101

**詳細シナリオ**:

3. 処理は続行する (エラーで止めない: ユーザーの意図かもしれないため)

**優先度**: Should Have

## エピック 5: vault 内容のエクスポート

### ストーリー 5.1: バックアップ目的の全エントリエクスポート 🔵

**信頼性**: 🔵 *PRD L231*

**私は** pq-diary から別ツール (Obsidian など) に乗り換えるかもしれないユーザー **として**
**`pq-diary export <DIR>` で全エントリを平文 MD で取り出したい**
**そうすることで** ロックインを避けられる

**関連要件**: REQ-501, REQ-502, REQ-503, REQ-504, REQ-505, NFR-004

**詳細シナリオ**:
1. ユーザーが `pq-diary export ~/backup/diary` を実行
2. システムが警告 `平文を ~/backup/diary に書き出します。続行しますか? [y/N]` を表示
3. ユーザーが `y` を入力
4. システムが全エントリを `~/backup/diary/YYYY-MM-DD-slug-id8.md` 形式で書き出し
5. システムが `Exported 42 entries to ~/backup/diary` を表示

**前提条件**:
- vault が存在
- 出力ディレクトリが存在し書き込み可能

**制約事項**:
- 出力 MD ファイルは YAML フロントマターでメタデータを保持
- タイトル衝突は UUID 8 桁プレフィックスで回避

**優先度**: Must Have

## エピック 6: 開発者体験の整合性

### ストーリー 6.1: ヘルプと実装の乖離を防ぐ 🔵

**信頼性**: 🔵 *2026-05-17 ヒアリング*

**私は** pq-diary の開発者 / メンテナ **として**
**ヘルプに出るコマンドが必ず実装されていることを保証したい**
**そうすることで** ユーザーが "出るのに動かない" 体験をしない

**関連要件**: REQ-701, REQ-702, REQ-703, REQ-704

**詳細シナリオ**:
1. 開発者が新スプリント実装後、DoD チェックリストを開く
2. 「CLI 整合性」セクションで全ヘルプ掲載コマンドが正常終了することを確認
3. CI が自動で smoke test を実行し、リグレッションを検出
4. 未実装スケルトン (legacy/daemon) はヘルプに出ないので、UX 上問題にならない

**前提条件**:
- CI 環境で smoke test スクリプトが実行可能

**優先度**: Must Have

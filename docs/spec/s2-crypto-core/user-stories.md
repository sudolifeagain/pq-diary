## エピック1: 鍵導出

### ストーリー 1.1: パスワードから鍵導出 🔵

**信頼性**: 🔵 *PRDセクション5.5より*

**私は** ユーザー **として**
**パスワードからArgon2idで安全に対称鍵を導出したい**
**そうすることで** パスワード1つでVaultの暗号化/復号に必要な鍵を得られる

**関連要件**: REQ-002

**詳細シナリオ**:
1. ユーザーがパスワードを入力する
2. システムがランダムなソルトを生成する（初回）または既存ソルトを読み込む
3. Argon2id (memory_cost_kb=65536, time_cost=3, parallelism=4) で32バイトの対称鍵を導出する
4. 導出された鍵は `ZeroizingKey` に格納される
5. 空パスワードの場合は `DiaryError::Password` を返す

**優先度**: Must Have

## エピック2: データ暗号化

### ストーリー 2.1: エントリデータの暗号化 🔵

**信頼性**: 🔵 *PRDセクション5.1より*

**私は** ユーザー **として**
**日記エントリをAES-256-GCMで暗号化・復号したい**
**そうすることで** エントリの機密性と改竄検知が保証される

**関連要件**: REQ-003

**詳細シナリオ**:
1. 平文エントリデータと32バイトの対称鍵を受け取る
2. `OsRng` でランダムな12バイト nonce を生成する
3. AES-256-GCM で暗号化し、暗号文と16バイト GCM タグを出力する
4. 復号時は同じ鍵と nonce を使用して平文を復元する
5. 暗号文が改竄されている場合は復号エラーを返す

**優先度**: Must Have

---

### ストーリー 2.2: 鍵カプセル化によるK_entry保護 🔵

**信頼性**: 🔵 *PRDセクション5.1 + FIPS 203より*

**私は** ユーザー **として**
**ML-KEM-768でエントリ鍵を安全にカプセル化したい**
**そうすることで** 量子コンピュータ耐性のある鍵交換が実現される

**関連要件**: REQ-004

**詳細シナリオ**:
1. ML-KEM-768 の鍵ペア（EncapsulationKey / DecapsulationKey）を生成する
2. EncapsulationKey でカプセル化し、暗号文と共有秘密を得る
3. DecapsulationKey で脱カプセル化し、同一の共有秘密を復元する
4. DecapsulationKey は Drop 時に zeroize される
5. 不正な暗号文に対しては脱カプセル化エラーを返す

**優先度**: Must Have

---

### ストーリー 2.3: エントリ改ざん検知 🔵

**信頼性**: 🔵 *PRDセクション5.1 + FIPS 204より*

**私は** ユーザー **として**
**ML-DSA-65でエントリの改ざんを検知したい**
**そうすることで** エントリが作成後に改竄されていないことを検証できる

**関連要件**: REQ-005, REQ-006

**詳細シナリオ**:
1. ML-DSA-65 の鍵ペア（VerifyingKey / SigningKey）を生成する
2. エントリデータに対して SigningKey で署名を生成する
3. VerifyingKey で署名を検証し、データの完全性を確認する
4. HMAC-SHA256 でデータのMACを計算し、追加の整合性検証を行う
5. 秘密鍵は Drop 時に zeroize される
6. 改竄されたデータまたは署名に対しては検証失敗を返す

**優先度**: Must Have

## エピック3: PQCフォーク

### ストーリー 3.1: フォーク作成とzeroizeパッチ 🔵

**信頼性**: 🔵 *PRDセクション5.3 + ヒアリングQ1/Q5より*

**私は** 開発者 **として**
**ml-kem / ml-dsa のフォークリポジトリを作成し、zeroizeパッチを適用したい**
**そうすることで** PQC秘密鍵のメモリ残存リスクを排除し、既知脆弱性を修正できる

**関連要件**: REQ-001

**詳細シナリオ**:
1. ml-kem リポジトリをフォークし、`pq-diary` ブランチを作成する
2. `DecapsulationKey` に zeroize 補完を追加する
3. ml-dsa リポジトリをフォークし、`pq-diary` ブランチを作成する
4. `s1_hat` / `s2_hat` / `t0_hat` / `A_hat` フィールドに zeroize を追加する
5. CVE-2026-22705 パッチを適用する
6. Cargo.toml で git 依存として参照する

**前提条件**:
- GitHub上にフォークリポジトリを作成する権限があること

**優先度**: Must Have

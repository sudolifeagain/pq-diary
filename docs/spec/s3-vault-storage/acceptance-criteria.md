## REQ-001: vault.pqd v4 バイナリフォーマット読み書き 🔵

**信頼性**: 🔵 *PRDセクション6.1より*

### Given（前提条件）
- vault.pqd v4フォーマットの仕様が定義されている
- バイナリシリアライズは自前実装で行う（外部クレート不使用）

### When（実行条件）
- vault.pqdヘッダの書き込みおよび読み込みを実行する

### Then（期待結果）
- マジックバイト `"PQDIARY\0"` が正しく書き込まれ、読み込み時に検証される
- スキーマバージョン `0x04` が正しく書き込まれ、読み込み時にチェックされる
- ペイロードサイズ、KDFソルト、Legacyソルト、検証トークン等の全フィールドが正確に読み書きされる

### テストケース

#### 正常系

- [ ] **TC-001-01**: 正しいマジックバイト+バージョンのヘッダ書き込み→読み込み一致 🔵
  - **入力**: マジックバイト `"PQDIARY\0"` + スキーマバージョン `0x04` でヘッダを書き込み
  - **期待結果**: 読み込んだヘッダのマジックバイトとバージョンが書き込み時と完全一致する

- [ ] **TC-001-02**: ペイロードサイズが正確 🔵
  - **入力**: 既知サイズのペイロードを含むvault.pqdを書き込み
  - **期待結果**: ヘッダのペイロードサイズフィールド (LE u32) が実際のペイロードサイズと一致する

- [ ] **TC-001-03**: KDFソルト/Legacyソルトの読み書き 🔵
  - **入力**: ランダムな32Bソルト2組をヘッダに書き込み
  - **期待結果**: 読み込んだKDFソルト (32B) とLegacyソルト (32B) が書き込み時と完全一致する

#### 異常系

- [ ] **TC-001-E01**: 不正マジックバイト → DiaryError::Vault 🔵
  - **入力**: マジックバイトが `"INVALID\0"` のファイルを読み込み
  - **期待結果**: `DiaryError::Vault` が返される

- [ ] **TC-001-E02**: 不正スキーマバージョン → DiaryError::Vault 🔵
  - **入力**: スキーマバージョンが `0x03` のファイルを読み込み
  - **期待結果**: `DiaryError::Vault` が返される

## REQ-002: エントリレコード読み書き 🔵

**信頼性**: 🔵 *PRDセクション6.2 + ADR-0002より*

### Given（前提条件）
- 有効なvault.pqdヘッダが存在する
- エントリレコードのバイナリ構造が定義されている

### When（実行条件）
- エントリレコードの書き込みおよび読み込みを実行する

### Then（期待結果）
- UUID、タイムスタンプ、IV、暗号文、署名、HMAC、legacyフィールド、添付ファイル予約フィールド、パディングが正確に読み書きされる
- レコード長=0で終端が正しく検出される

### テストケース

#### 正常系

- [ ] **TC-002-01**: エントリ1件の書き込み→読み込み一致（全フィールド） 🔵
  - **入力**: UUID、作成日時、更新日時、IV、暗号文、署名、HMAC、legacyフラグ、legacy鍵ブロック、パディングを含むエントリレコードを書き込み
  - **期待結果**: 読み込んだ全フィールドが書き込み時と完全一致する

- [ ] **TC-002-02**: エントリ複数件の書き込み→読み込み（順序保持） 🔵
  - **入力**: 3件のエントリレコードを順次書き込み
  - **期待結果**: 読み込んだ3件のエントリが書き込み順序を保持し、各フィールドが一致する

- [ ] **TC-002-03**: レコード長=0で正しく終端 🔵
  - **入力**: エントリレコードを書き込み後、レコード長=0の終端マーカーを書き込み
  - **期待結果**: 読み込み時にレコード長=0で正しく終端を検出し、それ以降のデータを読み込まない

- [ ] **TC-002-04**: 添付ファイル予約フィールドが0初期化 🔵
  - **入力**: エントリレコードを書き込み
  - **期待結果**: 添付ファイルカウント (u16) が0、添付ブロックオフセット (u64) が0で書き込まれている

#### 境界値

- [ ] **TC-002-B01**: 空vault（エントリ0件）→ 正常読み込み 🔵
  - **入力**: ヘッダのみで終端マーカー (レコード長=0) が続くvault.pqd
  - **期待結果**: エントリ0件として正常に読み込まれる

- [ ] **TC-002-B02**: 大量エントリ（100件）の読み書き 🔵
  - **入力**: 100件のエントリレコードを書き込み→読み込み
  - **期待結果**: 100件全てが正確に読み込まれ、各フィールドが一致する

### Then（期待結果）
- テンプレートレコードがエントリレコードと同じバイナリ構造で格納される
- レコードタイプフラグによりエントリとテンプレートが正しく区別される

### テストケース

#### 正常系

- [ ] **TC-003-01**: テンプレート書き込み→読み込み 🔵
  - **入力**: テンプレートレコードをレコードタイプフラグ付きで書き込み
  - **期待結果**: 読み込み時にテンプレートとして正しく識別され、全フィールドが一致する

- [ ] **TC-003-02**: エントリとテンプレートの混在読み込み 🔵
  - **入力**: エントリ2件 + テンプレート1件を書き込み
  - **期待結果**: 読み込み時にエントリとテンプレートが正しく分離される

## REQ-004: vault.toml デシリアライズ/シリアライズ 🔵

**信頼性**: 🔵 *PRDセクション3.2 + ヒアリングQ2より*

### Given（前提条件）
- VaultConfig構造体が serde derive で定義されている
- toml クレートでシリアライズ/デシリアライズを行う

### When（実行条件）
- vault.toml のシリアライズ/デシリアライズを実行する

### Then（期待結果）
- 全セクション (vault, access, git, git.privacy, argon2) が正しく読み書きされる
- タイムスタンプフィールドが含まれない

### テストケース

#### 正常系

- [ ] **TC-004-01**: VaultConfig のシリアライズ→デシリアライズ一致 🔵
  - **入力**: 全フィールドを設定したVaultConfigをシリアライズ→デシリアライズ
  - **期待結果**: デシリアライズ後のVaultConfigが元のオブジェクトと完全一致する

- [ ] **TC-004-02**: デフォルト値の生成 🔵
  - **入力**: VaultConfig::default() を実行
  - **期待結果**: name="private", schema_version=4, policy="none", memory_cost_kb=65536, time_cost=3, parallelism=4 等のデフォルト値が設定される

- [ ] **TC-004-03**: 全フィールドの読み取り（vault/access/git/argon2） 🔵
  - **入力**: PRD仕様のサンプルvault.tomlをデシリアライズ
  - **期待結果**: vault.name, access.policy, git.author_name, git.author_email, git.commit_message, git.privacy.timestamp_fuzz_hours, git.privacy.extra_padding_bytes_max, argon2.memory_cost_kb, argon2.time_cost, argon2.parallelism が正しく読み取られる

#### 異常系

- [ ] **TC-004-E01**: 不正TOML → DiaryError::Config 🔵
  - **入力**: 構文エラーを含むTOML文字列をデシリアライズ
  - **期待結果**: `DiaryError::Config` が返される

- [ ] **TC-006-02**: vault list で全Vault一覧 🔵
  - **入力**: "private" と "notes" の2つのVaultを初期化後、vault list を実行

- [ ] **TC-006-03**: -v オプションでVault切り替え 🔵

## REQ-007: 検証トークン生成・検証 🔵

**信頼性**: 🔵 *PRDセクション6.1より*

### Given（前提条件）
- S2のCryptoEngine（AES-256-GCM暗号化/復号）が利用可能である
- パスワードから導出された対称鍵が利用可能である

### When（実行条件）
- 検証トークンの生成、格納、読み出し、検証を実行する

### Then（期待結果）
- 32Bランダム平文がAES-GCMで暗号化されてvault.pqdヘッダに格納される
- unlock時に正しいパスワードで復号→一致確認が成功する
- 不正パスワードでは検証が失敗する

### テストケース

#### 正常系

- [ ] **TC-007-01**: 検証トークン生成→格納→読み出し→検証成功 🔵
  - **入力**: パスワードから鍵を導出し、検証トークンを生成→vault.pqdヘッダに格納→同じパスワードで読み出し→検証
  - **期待結果**: 検証が成功する

#### 異常系

- [ ] **TC-007-02**: 不正パスワード → 検証失敗 🔵
  - **入力**: パスワードAでトークン生成→パスワードBで復号→検証
  - **期待結果**: 復号結果が元の平文と不一致となり、検証が失敗する

## REQ-008: ランダムパディング 🔵

**信頼性**: 🔵 *PRDセクション6.1, 9.1より*

### Given（前提条件）
- ランダムパディングの範囲が定義されている（ファイル末尾: 512〜4096B、エントリ: 0〜255B）

### When（実行条件）
- vault.pqd書き込みおよびエントリレコード書き込みを実行する

### Then（期待結果）
- ファイル末尾に512〜4096Bのランダムパディングが付与される
- 各エントリレコードに0〜255Bのランダムパディングが付与される
- 同一データの2回書き込みでファイルサイズが異なる（ランダム性の検証）

### テストケース

#### 正常系

- [ ] **TC-008-01**: ファイル末尾パディングが512〜4096B 🔵
  - **入力**: vault.pqdを書き込み、ファイル末尾のパディングサイズを測定
  - **期待結果**: パディングサイズが512B以上4096B以下である

- [ ] **TC-008-02**: エントリパディングが0〜255B 🔵
  - **入力**: エントリレコードを書き込み、パディングサイズを読み取り
  - **期待結果**: パディング長フィールド (1B) の値が0〜255の範囲であり、実際のパディングバイト数と一致する

- [ ] **TC-008-03**: 同一データの2回書き込みでファイルサイズが異なる（ランダムパディング） 🔵
  - **入力**: 同一のエントリデータでvault.pqdを2回書き込み
  - **期待結果**: 2つのファイルサイズが異なる（ランダムパディングにより同一データでもファイルサイズが変動する）

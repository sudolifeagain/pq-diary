## エピック1: Vault初期化

### ストーリー 1.1: 新しいVaultを作成 🔵

**信頼性**: 🔵 *PRDセクション3.1, 6.1より*

**私は** ユーザー **として**
**新しいVaultを初期化し、暗号化された日記保管領域を作成したい**
**そうすることで** パスワードで保護された安全な日記ストレージを使い始められる

**関連要件**: REQ-006, REQ-007, REQ-008

**詳細シナリオ**:
1. ユーザーが `pq-diary vault init <name>` コマンドを実行する
2. システムが `~/.pq-diary/vaults/<name>/` ディレクトリを作成する
3. ML-KEM-768 / ML-DSA-65 の鍵ペアを生成する
4. パスワードからArgon2idで対称鍵を導出する
5. 検証トークン（32Bランダム平文をAES-GCMで暗号化）を生成する
6. vault.pqd v4ヘッダを書き込み、ランダムパディングを付与する
7. デフォルト値で vault.toml を生成する
8. entries/ ディレクトリを作成する

**前提条件**:
- S2のCryptoEngine（鍵生成・暗号化機能）が利用可能であること

**優先度**: Must Have

## エピック2: Vaultフォーマット

### ストーリー 2.1: vault.pqdの読み書き 🔵

**信頼性**: 🔵 *PRDセクション6.1, 6.2より*

**私は** システム **として**
**vault.pqd v4バイナリフォーマットを正確に読み書きしたい**
**そうすることで** 暗号化された日記エントリを永続化し、安全に復元できる

**関連要件**: REQ-001, REQ-002

**詳細シナリオ**:
1. ヘッダ書き込み: マジックバイト、スキーマバージョン、フラグ、ペイロードサイズ、KDFソルト、Legacyソルト、検証トークン、鍵情報を自前バイナリシリアライズで書き込む
2. エントリレコード書き込み: UUID、タイムスタンプ、IV、暗号文、署名、HMAC、legacyフィールド、添付ファイル予約フィールド（0初期化）、パディングを書き込む
3. ファイル末尾にランダムパディング（512〜4096B）を付与する
4. 読み込み時はマジックバイトとスキーマバージョンを検証する
5. レコード長=0で終端を検出する
6. `strings vault.pqd` でフィールド名・アルゴリズム名が露出しないことを保証する

**優先度**: Must Have

---

### ストーリー 2.2: テンプレート格納 🔵

**信頼性**: 🔵 *ADR-0005より*

**私は** ユーザー **として**
**日記テンプレートをVault内に暗号化して格納したい**
**そうすることで** テンプレートも日記エントリと同じセキュリティレベルで保護される

**関連要件**: REQ-003

**詳細シナリオ**:
1. テンプレートレコードはエントリレコードと同じバイナリ構造を持つ
2. レコードタイプフラグでエントリとテンプレートを区別する
3. vault.pqd内に暗号化された状態で格納される
4. 読み込み時にレコードタイプに応じてエントリとテンプレートを分離する

**優先度**: Must Have

## エピック3: 設定管理

### ストーリー 3.1: vault.toml管理 🔵

**信頼性**: 🔵 *PRDセクション3.2 + ヒアリングQ2より*

**私は** ユーザー **として**
**Vault固有の設定をvault.tomlで管理したい**
**そうすることで** Vaultごとに異なるArgon2idパラメータやGit設定を柔軟に設定できる

**関連要件**: REQ-004

**詳細シナリオ**:
1. VaultConfig構造体を serde + toml でデシリアライズ/シリアライズする
2. `[vault]` セクション: name, schema_version
3. `[access]` セクション: policy
4. `[git]` セクション: author_name, author_email, commit_message
5. `[git.privacy]` セクション: timestamp_fuzz_hours, extra_padding_bytes_max
6. `[argon2]` セクション: memory_cost_kb, time_cost, parallelism
7. vault.tomlが存在しない場合はデフォルト値で自動生成する
8. タイムスタンプフィールドを含めない（PRD v4.0変更）

**優先度**: Must Have

---

### ストーリー 3.2: config.toml管理 🔵

**信頼性**: 🔵 *PRDセクション3.3 + ヒアリングQ2より*

**私は** ユーザー **として**
**アプリケーション全体の設定をconfig.tomlで管理したい**
**そうすることで** デフォルトVaultやデーモン設定を一元管理できる

**関連要件**: REQ-005

**詳細シナリオ**:
1. AppConfig構造体を serde + toml でデシリアライズ/シリアライズする
2. `[defaults]` セクション: vault（デフォルトVault名）
3. `[daemon]` セクション: socket_dir, timeout_secs
4. config.tomlが存在しない場合はデフォルト値で自動生成する

**優先度**: Must Have

## エピック4: マルチVault

### ストーリー 4.1: 複数Vaultの管理 🔵

**信頼性**: 🔵 *PRDセクション3.1より*

**私は** ユーザー **として**
**用途に応じて複数のVaultを作成・切り替えて使いたい**
**そうすることで** 「private」「notes」などの目的別にジャーナルを分離して管理できる

**関連要件**: REQ-006

**詳細シナリオ**:
1. `~/.pq-diary/vaults/` 配下に複数のVaultディレクトリを作成できる
2. `vault list` で全Vault一覧を表示する
3. `-v <name>` オプションで操作対象のVaultを切り替える
4. config.toml の `[defaults].vault` でデフォルトVaultを設定する
5. 各Vaultは独立した vault.pqd / vault.toml / entries/ を持つ

**前提条件**:
- config.toml によるデフォルトVault設定が利用可能であること

**優先度**: Must Have

# S7 Access Control + Claude ヒアリング記録

**作成日**: 2026-04-10
**ヒアリング実施**: step4 既存情報ベースの差分ヒアリング

## ヒアリング目的

PRD v4.0（section 4.4）と既存実装（policy.rs stub, --claude flag, VaultCommands enum）を精査し、S7実装のスコープ・設計判断の不明点を明確化するためのヒアリングを実施しました。

## 質問と回答

### Q1: vault create のS7実装範囲

**質問日時**: 2026-04-10
**カテゴリ**: 影響範囲
**背景**: PRDでは `vault create <n> [--policy <POLICY>]` が定義されているが、現在の実装ではvault createはSprint 2のスタブ。initコマンドが既にvault作成を担っているため、S7でのスコープを確認する必要があった。

**回答**: S7でvault createも実装する

**信頼性への影響**:
- REQ-030〜REQ-032 の信頼性が 🟡 → 🔵 に向上
- vault list/delete の実装もS7スコープに含まれることが確定

### Q2: write_onlyポリシーでのsearch操作

**質問日時**: 2026-04-10
**カテゴリ**: 既存設計確認
**背景**: PRDではwrite_onlyは「書き込み・sync・deleteのみ」と定義。searchは復号を伴うためread系だが、Claudeのユースケースでは検索できると便利な可能性があった。

**回答**: searchは拒否（PRD通り）

**信頼性への影響**:
- REQ-010 の操作分類が確定（🔵）
- PRD準拠の方針が確認され、全体的な設計判断の信頼性が向上

---

### Q10: vault CRUD仕様一括確認

**質問日時**: 2026-04-10
**カテゴリ**: 既存設計確認
**背景**: 残りの黄信号項目を一括で確認。7項目をまとめて提示。

**回答**: 全て問題なし
1. AccessSection.policy を String → AccessPolicy enum に置換 ✅
2. 既存vault.toml（policy="none"文字列）との後方互換性維持 ✅
3. 同名Vault存在時はvault createエラー ✅
4. vault.toml書き込みはアトミック操作（temp+rename） ✅
5. 無効ポリシー値はデシリアライズエラー（フォールバックしない） ✅
6. Vault名バリデーション（空文字列拒否、パストラバーサル拒否） ✅
7. vaults/ディレクトリ未存在時はvault createで自動作成 ✅

**信頼性への影響**:
- REQ-003, REQ-032, REQ-404, EDGE-001〜005, EDGE-101〜103 がすべて 🟡 → 🔵

## ヒアリング結果サマリー

### 確認できた事項
- S7スコープ: ポリシー評価 + vault CRUD(create/list/policy/delete) + Claude全コマンド対応
- ポリシー型: Rust enum (None/WriteOnly/Full) + serde対応
- write_only操作分類: PRD準拠（read系拒否）
- パスワード制限: なし（既存3段階をそのまま使用）
- full警告: ポリシー設定時のみ
- エラーメッセージ: 詳細（Vault名・ポリシー・必要権限）
- vault delete: オプションzeroize対応（--zeroize）
- PW要否: createのみ必要、list/policy/deleteは不要
- vault list: 名前+ポリシーのみ
- 改ざん時: エラー返却（フォールバックなし）

### 追加/変更要件
- **追加**: --claudeの全コマンド対応（PRDの4コマンドから拡張）
- **追加**: vault create/list/delete（元々別スプリント予定）
- **追加**: AccessPolicy enum化（既存String型からの移行）
- **追加**: REQ-037 vault delete --zeroize オプション

### 残課題
- なし（全項目確認済み）

## エピック1: Git Setup（git-init, git-status）

### ストーリー 1.1: Vaultのgit初期化 🔵

**信頼性**: 🔵 *PRD 4.1 + PRD 11.1 + ヒアリングQ1「S8フルスコープ」+ Q2「git-init時にauthor生成」*

**私は** pq-diaryユーザー **として**
**VaultをGitリポジトリとして初期化し、リモートと同期する準備をしたい**
**そうすることで** 複数デバイス間で暗号化された日記を安全に同期できる

**関連要件**: REQ-001, REQ-002, REQ-003, REQ-004, REQ-005

**詳細シナリオ**:
1. ユーザーが `pq-diary git-init --remote git@github.com:user/vault.git` を実行
2. Vaultディレクトリ内に `.git` が初期化される
3. `.gitignore` が生成され、`entries/*.md` が除外される
4. ランダムな `author_email`（例: `a3f7c901@localhost`）が生成される
5. vault.toml の `[git]` セクションが更新される（author_name: "pq-diary", author_email: 生成値）
6. リモート `origin` が設定される
7. パスワード入力は不要

**前提条件**:
- Vaultが作成済み
- gitがインストール済み
- `.git` が未初期化

**制約事項**:
- パスワード不要
- 既にgit初期化済みの場合はエラー

**優先度**: Must Have

## エピック2: Git Sync（git-push, git-pull, git-sync + プライバシー）

### ストーリー 2.1: プライバシー保護付きgit-push 🔵

**信頼性**: 🔵 *PRD 11.3 + ADR-0006 + ヒアリングQ3「pushはPW必要」*

**私は** pq-diaryユーザー **として**
**Vault変更をプライバシー保護付きでリモートにpushしたい**
**そうすることで** Gitコミット履歴から作業パターンや個人情報が推測されることを防げる

**関連要件**: REQ-010, REQ-011, REQ-012, REQ-013, REQ-014, REQ-015, REQ-016, REQ-017

**詳細シナリオ**:
1. ユーザーが `pq-diary git-push` を実行
2. パスワードの入力を求められる
3. vault.pqd に追加パディング（0〜extra_padding_bytes_max）が付与される
4. `vault.pqd` と `vault.toml` がステージングされる
5. コミットauthor: `pq-diary <a3f7c901@localhost>`
6. コミットメッセージ: vault.toml の commit_message（デフォルト "Update vault"）
7. GIT_AUTHOR_DATE / GIT_COMMITTER_DATE がファジングされる（単調増加保証）
8. `git push` が実行される

**前提条件**:
- Vaultがgit初期化済み
- リモートが設定済み

**制約事項**:
- パスワード必要（追加パディング付き再書き込み）
- extra_padding_bytes_max=0 の場合はパディング無効

**優先度**: Must Have

---

### ストーリー 2.2: git-pullとエントリマージ 🔵

**信頼性**: 🔵 *PRD 11.4 + ヒアリングQ3「pullはPW必要」+ Q4「UUID+content_hmac」*

**私は** pq-diaryユーザー **として**
**リモートの変更をpullし、ローカルの変更とマージしたい**
**そうすることで** 複数デバイスでの変更が統合される

**関連要件**: REQ-020, REQ-021, REQ-022, REQ-023, REQ-024, REQ-027, REQ-028

**詳細シナリオ**:
1. ユーザーが `pq-diary git-pull` を実行
2. パスワードの入力を求められる
3. リモートからfetchされる
4. ローカルとリモートのvault.pqdがそれぞれ復号される
5. UUID + content_hmac でエントリ単位の3-wayマージが実行される
6. リモートにのみ存在する新規エントリが追加される
7. リモートで削除されたエントリが削除される
8. マージ結果がvault.pqdに再暗号化・アトミック書き込みされる

**前提条件**:
- Vaultがgit初期化済み
- リモートが設定済み

**制約事項**:
- パスワード必要（vault.pqd復号のため）

**優先度**: Must Have

---

### ストーリー 2.3: git-syncワンコマンド同期 🔵

**信頼性**: 🔵 *ヒアリングQ3「sync = pull → push」*

**私は** pq-diaryユーザー **として**
**ワンコマンドでpullとpushを連続実行したい**
**そうすることで** 同期操作を簡単に行える

**関連要件**: REQ-030, REQ-031

**詳細シナリオ**:
1. ユーザーが `pq-diary git-sync` を実行
2. パスワードの入力を求められる
3. まず `git-pull`（マージ含む）が実行される
4. 次に `git-push`（プライバシー保護付き）が実行される

**前提条件**:
- Vaultがgit初期化済み
- リモートが設定済み

**制約事項**:
- パスワード必要
- pullが失敗した場合、pushは実行しない

**優先度**: Must Have

## エピック3: Merge & Conflict Resolution（3-wayマージ, コンフリクト解決）

### ストーリー 3.1: コンフリクト対話式解決 🔵

**信頼性**: 🔵 *PRD 11.4 + ヒアリングQ3*

**私は** pq-diaryユーザー **として**
**ローカルとリモートで同じエントリが変更された場合、どちらを採用するか対話的に選択したい**
**そうすることで** 重要な変更を失わずにマージできる

**関連要件**: REQ-025

**詳細シナリオ**:
1. `git-pull` 実行中にUUID `abc123` のエントリがローカル・リモート双方で変更を検出
2. コンフリクトプロンプトが表示される：
   - ローカル版の概要（タイトル、更新日時）
   - リモート版の概要（タイトル、更新日時）
   - 選択肢: [L]ocal / [R]emote / [S]kip
3. ユーザーが選択すると、選択された版でマージが続行される

**前提条件**:
- 同一UUIDエントリの content_hmac がローカルとリモートで異なる

**優先度**: Must Have

---

### ストーリー 3.2: --claude時のコンフリクト自動解決 🔵

**信頼性**: 🔵 *ヒアリングQ3「--claudeコンフリクト: ローカル勝ち」*

**私は** Claude Codeを使うユーザー **として**
**--claude実行時のコンフリクトを自動的にローカル優先で解決したい**
**そうすることで** 非対話環境でも同期が完了する

**関連要件**: REQ-026

**詳細シナリオ**:
1. `pq-diary --claude git-pull` 実行中にコンフリクトを検出
2. 対話プロンプトなしでローカル側が自動的に採用される
3. マージが正常に完了する

**前提条件**:
- `--claude` フラグが指定されている

**優先度**: Must Have

---

### ストーリー 3.3: エッジケース対応マージ 🔵

**信頼性**: 🔵 *ヒアリングQ4 確定*

**私は** pq-diaryユーザー **として**
**空のリモートや空のローカルの場合でもマージが正しく動作してほしい**
**そうすることで** 初回同期や新デバイスセットアップがスムーズに行える

**関連要件**: REQ-027, EDGE-004, EDGE-005

**詳細シナリオ**:
1. 新デバイスで `git-pull` を実行（ローカルにエントリなし）
2. リモートの全エントリがローカルに追加される
3. vault.pqd が再暗号化・書き込みされる

**前提条件**:
- Vaultがgit初期化済み

**優先度**: Must Have

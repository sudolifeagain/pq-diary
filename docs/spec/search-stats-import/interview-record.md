## ヒアリング目的

backlog.md S6 の記述は概要レベルのため、search/stats/import の具体仕様を明確化する差分ヒアリングを実施した。requirements.md v4.0 にはこれらのコマンドの詳細定義が存在しないことが確認された。

## 質問と回答

### Q1: search コマンドの検索対象

**カテゴリ**: 未定義部分詳細化
**背景**: backlog には「復号後インメモリ正規表現grep」とあるが、検索対象フィールドが未定義

**回答**: エントリ本文 (body)、タイトル (title)、タグ (tags)、テンプレート本文 — 全4フィールドを対象

**信頼性への影響**: REQ-B03 の信頼性が 🔴 → 🔵 に向上

---

### Q2: import のソース形式

**カテゴリ**: 未定義部分詳細化
**背景**: backlog には「プレーンMD一括取り込み」とあるが、Obsidian 固有の処理が必要か不明

**回答**: Obsidian Vault + プレーン Markdown の両方を対応

**信頼性への影響**: REQ-D01, REQ-D08 の信頼性が 🔵 に確定

---

### Q3: stats コマンドの出力形式

**カテゴリ**: 未定義部分詳細化
**背景**: backlog には「執筆頻度, 文字数推移, タグ分布」とあるが出力形式が未定義

**回答**: テキスト表示 + JSON 出力 + 日別ヒートマップ — 全3形式を対応

**信頼性への影響**: REQ-C03, REQ-C04, REQ-C05 の信頼性が 🔵 に確定

---

### Q4: 技術的負債と新機能の優先順位

**カテゴリ**: 優先順位
**背景**: S6 のスコープに技術的負債 13 件と新機能 3 コマンドが混在

**回答**: 負債先行 — CRITICAL/HIGH のセキュリティ負債を先に修正し、その後に新機能を実装

**信頼性への影響**: Phase A → Phase B/C/D の実装順序が 🔵 に確定

---

### Q5: search のコンテキスト行数

**カテゴリ**: 未定義部分詳細化
**背景**: backlog には「前後N行」とあるがデフォルト値が未定義

**回答**: 前後2行（`--context N` で変更可能）

**信頼性への影響**: REQ-B05 の信頼性が 🔵 に確定

---

### Q6: import のタイトル抽出方法

**カテゴリ**: 未定義部分詳細化
**背景**: Markdown ファイルからエントリタイトルをどう決定するか未定義

**回答**: ファイル名優先（拡張子除去）。YAML frontmatter に title があればそちらを優先

**信頼性への影響**: REQ-D03 の信頼性が 🔵 に確定

## ヒアリング結果サマリー

### 確認できた事項
- search の検索対象は body + title + tags + template body の全4フィールド
- import は Obsidian Vault と plain Markdown の両方に対応
- stats は text + JSON + heatmap の3形式出力
- 実装順序: 負債修正 → 新機能
- search のデフォルトコンテキスト: 前後2行
- import のタイトル: ファイル名優先、frontmatter があればそちら

### 追加/変更要件
- REQ-C05 (heatmap) — backlog に未記載だがユーザー要望として追加
- REQ-B07 (--count) — grep 慣例に基づき追加提案

- import の大量ファイル時のプログレス表示の要否

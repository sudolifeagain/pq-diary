# 検索 + 統計 + インポート + 技術的負債 要件定義書

## 概要

Sprint 6 では、pq-diary に全文検索 (search)、統計情報表示 (stats)、外部 Markdown 一括取り込み (import) の3つの新コマンドを追加する。加えて、S1-S5 コードレビューで特定された CRITICAL/HIGH/MEDIUM のセキュリティ技術的負債 13 件を修正する。負債修正を先行し、その後に新機能を実装する。

## 機能要件（EARS記法）

**【信頼性レベル凡例】**:
- 🔵 **青信号**: PRD・設計文書・ユーザヒアリングを参考にした確実な要件
- 🟡 **黄信号**: PRD・設計文書・ユーザヒアリングから妥当な推測による要件

### Phase A: 技術的負債修正

#### 制約要件 (CRITICAL)

- REQ-A01: vault reader は可変長フィールド (`kem_encrypted_sk`, `dsa_encrypted_sk`, `ciphertext`, `signature`, `legacy_key_block`) の読み取り時に、各フィールドサイズが 16 MiB を超える場合 `DiaryError::Vault` を返さなければならない 🔵 *S3レビュー C-1/C-2*
- REQ-A02: vault reader は `record_len` が 16 MiB を超える場合 `DiaryError::Vault` を返さなければならない 🔵 *S3レビュー C-1/C-2*

#### 制約要件 (HIGH)

- REQ-A03: `VaultManager::init_vault` は空パスワード (`password.is_empty()`) を受け取った場合 `DiaryError::Password` を返さなければならない 🔵 *S3レビュー H-3*
- REQ-A04: vault writer の全 `usize` → `u32`/`u8` 変換は `u32::try_from()` / `u8::try_from()` を使い、オーバーフロー時に `DiaryError::Vault` を返さなければならない 🔵 *S3レビュー H-4*

#### 制約要件 (MEDIUM)

- REQ-A05: `write_vault` はアトミック書き込み (一時ファイルへ書き込み後に `rename`) で実装しなければならない 🔵 *S3レビュー M-6*
- REQ-A06: vim/nvim 起動時のオプションに `nowritebackup` と `viminfo=''` (nvim: `shada=''`) を追加しなければならない 🔵 *S4レビュー M-7*
- REQ-A07: `PQ_DIARY_PASSWORD` 環境変数は読み取り後に `std::env::remove_var` で削除しなければならない 🔵 *S4レビュー M-8*
- REQ-A08: 一時ファイル作成時の Unix パーミッションは 0o600 でなければならない 🔵 *S4レビュー M-9*
- REQ-A09: `hmac_util::compute` はエラー時にゼロ MAC を返すのではなく `Result<[u8; 32], DiaryError>` を返さなければならない 🔵 *S2レビュー M-5*
- REQ-A10: `EntryPlaintext` は `Zeroize` / `ZeroizeOnDrop` を derive しなければならない 🔵 *S5レビュー H-1*
- REQ-A11: `list_entries_with_body` の中間 `Vec<(EntryMeta, String)>` は `Zeroizing` でラップしなければならない 🔵 *S5レビュー H-2*
- REQ-A12: Win32 コンソール API の `unsafe` 使用について ADR を作成しなければならない 🟡 *S5レビュー M-7, ADR形式は推測*
- REQ-A13: CLI の vault 操作ボイラープレートを drop guard パターンで抽出しなければならない 🟡 *S5レビュー L-2, 設計詳細は推測*

### Phase B: 新機能 — search コマンド

#### 通常要件

- REQ-B01: システムは `pq-diary search <PATTERN>` コマンドを提供しなければならない 🔵 *backlog.md S6*
- REQ-B02: `PATTERN` は Rust `regex` クレートの正規表現構文で解釈しなければならない 🟡 *backlog「正規表現grep」から推測*
- REQ-B03: 検索はエントリ本文 (body)、タイトル (title)、タグ (tags)、テンプレート本文を対象としなければならない 🔵 *ユーザヒアリング 2026-04-08*
- REQ-B04: 検索結果はマッチしたエントリごとに `{id_prefix} {date} {title}` を表示し、マッチ行とその前後のコンテキスト行を表示しなければならない 🟡 *backlog + grep慣例から推測*

#### 条件付き要件

- REQ-B05: `--context N` フラグが指定された場合、マッチ行の前後 N 行を表示しなければならない。デフォルトは 2 行である 🔵 *ユーザヒアリング 2026-04-08*
- REQ-B06: `--tag TAG` フラグが指定された場合、指定タグ（前方一致）でフィルタした上で検索しなければならない 🟡 *既存 list コマンドとの一貫性から推測*
- REQ-B07: `--count` フラグが指定された場合、マッチしたエントリ数のみを表示しなければならない 🟡 *grep -c 慣例から推測*

#### 制約要件

- REQ-B08: 検索は全エントリを復号しインメモリで実行しなければならない（vault.pqd にインデックスを追加しない） 🔵 *backlog「復号後インメモリ」*
- REQ-B09: 復号した平文は検索完了後に zeroize しなければならない 🔵 *requirements.md 設計原則 1.2-2*

### Phase C: 新機能 — stats コマンド

#### 通常要件

- REQ-C01: システムは `pq-diary stats` コマンドを提供しなければならない 🔵 *backlog.md S6*
- REQ-C02: stats は以下の統計情報を表示しなければならない: エントリ総数、タグ総数、最初/最後のエントリ日付、執筆頻度（直近30日のアクティブ日数）、文字数統計（合計/平均/最大）、タグ分布（上位10タグ） 🟡 *backlog「執筆頻度, 文字数推移, タグ分布」から推測*
- REQ-C03: デフォルト出力はテキスト形式で、人間が読みやすいフォーマットでなければならない 🔵 *ユーザヒアリング 2026-04-08*

#### 条件付き要件

- REQ-C04: `--json` フラグが指定された場合、全統計データを JSON 形式で stdout に出力しなければならない 🔵 *ユーザヒアリング 2026-04-08*
- REQ-C05: `--heatmap` フラグが指定された場合、直近1年の日別執筆頻度を ASCII ヒートマップで表示しなければならない 🔵 *ユーザヒアリング 2026-04-08*

#### 制約要件

- REQ-C06: stats は全エントリのメタデータおよび本文を復号して集計しなければならない（文字数集計のため） 🟡 *文字数推移を計算するには本文復号が必要、と推測*

---

### Phase D: 新機能 — import コマンド

#### 通常要件

- REQ-D01: システムは `pq-diary import <DIR>` コマンドを提供しなければならない 🔵 *backlog.md S6*
- REQ-D02: import は指定ディレクトリ内の `.md` ファイルを再帰的に走査しなければならない 🟡 *「一括取り込み」から推測*
- REQ-D03: 各 `.md` ファイルについて、ファイル名（拡張子除去）をタイトルとして使用しなければならない。ただし YAML frontmatter に `title` フィールドが存在する場合はそちらを優先する 🔵 *ユーザヒアリング 2026-04-08*
- REQ-D04: ファイル本文中の `[[wiki-link]]` 表記を `[[タイトル]]` 形式に変換しなければならない 🔵 *backlog.md S6*
- REQ-D05: ファイル本文中の `#ネスト/タグ` 表記を pq-diary のネストタグ形式に変換しなければならない 🔵 *backlog.md S6*
- REQ-D06: YAML frontmatter の `tags` フィールドが存在する場合、エントリのタグとして取り込まなければならない 🟡 *Obsidian 慣例から推測*

#### 条件付き要件

- REQ-D07: `--dry-run` フラグが指定された場合、実際のインポートを行わずに対象ファイル数と変換結果のプレビューを表示しなければならない 🟡 *安全性のため推測*
- REQ-D08: Obsidian Vault 形式のディレクトリの場合、`.obsidian/` ディレクトリとその他の非 `.md` ファイルを自動的にスキップしなければならない 🔵 *ユーザヒアリング「Obsidian Vault」対応*

#### 通常要件 (結果表示)

- REQ-D09: インポート完了後にサマリーを表示しなければならない: 取り込み件数、スキップ件数、変換されたリンク数、変換されたタグ数 🔵 *backlog「インポート結果サマリー表示」*

#### 制約要件

- REQ-D10: インポートはバッチ書き込み（全エントリを一度に `write_vault`）で行い、1エントリごとの書き込みを避けなければならない 🟡 *パフォーマンス観点から推測*
- REQ-D11: 読み込んだ Markdown ファイルの内容バッファは使用後に zeroize しなければならない 🔵 *requirements.md 設計原則 1.2-2*
- REQ-D12: import は元ファイルを削除してはならない（読み取り専用） 🟡 *安全性のため推測*

## 非機能要件

### パフォーマンス

- NFR-001: search コマンドは 100 エントリに対して 1 秒以内に結果を返さなければならない 🟡 *list < 500ms 基準の2倍として推測*
- NFR-002: stats コマンドは 100 エントリに対して 1 秒以内に結果を返さなければならない 🟡 *同上*
- NFR-003: import コマンドは 100 ファイルに対して 10 秒以内に完了しなければならない（暗号化のオーバーヘッドを含む） 🟡 *推測*

### セキュリティ

- NFR-101: 復号した平文は操作完了後に即座に zeroize しなければならない 🔵 *requirements.md 1.2-2*
- NFR-102: search パターンがシェル履歴に残ることについてドキュメントで注意喚起しなければならない 🟡 *セキュリティ観点から推測*

### ユーザビリティ

- NFR-201: search の出力は grep に慣れたユーザーが直感的に理解できるフォーマットでなければならない 🟡 *推測*
- NFR-202: stats のヒートマップは端末幅 80 文字以上で正しく表示されなければならない 🟡 *推測*

## エッジケース

### エラー処理

- EDGE-001: search パターンが無効な正規表現の場合、明確なエラーメッセージを表示して終了しなければならない 🟡 *regex クレートのエラーハンドリングから推測*
- EDGE-002: import ディレクトリが存在しない場合、エラーメッセージを表示しなければならない 🟡 *推測*
- EDGE-003: import ディレクトリに `.md` ファイルが 0 件の場合、「No Markdown files found」と表示しなければならない 🟡 *推測*

### 境界値

- EDGE-101: 空の vault (エントリ 0 件) に対する search は「No matches found」を表示しなければならない 🟡 *推測*
- EDGE-102: 空の vault に対する stats は基本情報（エントリ数: 0）のみを表示しなければならない 🟡 *推測*
- EDGE-103: import で同名タイトルのエントリが既に存在する場合、スキップして警告を表示しなければならない 🔴 *未定義、推測*

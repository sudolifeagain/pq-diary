## タスク概要

`core/src/lib.rs` の `DiaryCore` 構造体をファサードとして拡張し、エントリ操作の公開 API を提供する。`vault_path`、`engine` (CryptoEngine)、`config` (VaultConfig) をフィールドとして保持し、`new()` / `unlock()` / `lock()` でライフサイクルを管理する。`new_entry()` / `list_entries()` / `get_entry()` / `update_entry()` / `delete_entry()` は `entry.rs` の関数に委譲する。Sprint 4 以降の CLI 層がこのファサードのみを通じて core 層にアクセスする。

## 実装詳細

### 1. DiaryCore 構造体拡張 🔵

**信頼性**: 🔵 *PRD §2.3, architecture.md §5.5より*

```rust
pub struct DiaryCore {
    vault_path: PathBuf,
    engine: Option<CryptoEngine>,
    config: VaultConfig,
}
```

- `vault_path`: vault.pqd ファイルへのパス
- `engine`: unlock 後に Some、lock 後に None。エントリ操作メソッドは engine が Some であることを検証
- `config`: vault.toml から読み込んだ設定

**実装ファイル**: `core/src/lib.rs`

### 2. ライフサイクル管理メソッド 🔵

**信頼性**: 🔵 *PRD §2.3, REQ-092より*

```rust
impl DiaryCore {
    pub fn new(vault_path: &str) -> Result<Self, DiaryError>;
    pub fn unlock(&mut self, password: SecretString) -> Result<(), DiaryError>;
    pub fn lock(&mut self);
}
```

- `new()`: vault.toml をパースして `VaultConfig` を取得、`engine` は None で初期化
- `unlock()`: パスワードから Argon2id で対称鍵を導出し `CryptoEngine` を構築
- `lock()`: `engine` を `None` に戻す (CryptoEngine の Drop で zeroize が実行される)

**実装ファイル**: `core/src/lib.rs`

### 3. エントリ操作メソッド (委譲) 🔵

**信頼性**: 🔵 *architecture.md §5.5, types.rs DiaryCore定義より*

```rust
impl DiaryCore {
    pub fn new_entry(&self, title: &str, body: &str, tags: Vec<String>) -> Result<String, DiaryError>;
    pub fn list_entries(&self, query: Option<&str>) -> Result<Vec<EntryMeta>, DiaryError>;
    pub fn get_entry(&self, id: &str) -> Result<(EntryRecord, EntryPlaintext), DiaryError>;
    pub fn update_entry(&self, id: &str, plaintext: &EntryPlaintext) -> Result<(), DiaryError>;
    pub fn delete_entry(&self, id: &str) -> Result<(), DiaryError>;
}
```

各メソッドの共通パターン:
1. `self.engine` が `Some` であることを確認 (None なら `DiaryError::Entry("Vaultがロック中です")`)
2. 必要に応じて `IdPrefix::new(id)` でバリデーション
3. `entry.rs` の対応する関数に委譲

**実装ファイル**: `core/src/lib.rs`

### 4. モジュールエクスポート更新 🔵

**信頼性**: 🔵 *Rust モジュールシステムの標準パターンより*

`lib.rs` の公開モジュール宣言を更新し、以下の型・関数を外部クレートから利用可能にする:
- `EntryPlaintext`, `EntryMeta`, `Tag`, `IdPrefix`
- `DiaryCore` (ファサード)

**実装ファイル**: `core/src/lib.rs`

## テスト要件

### E2E テスト: フルライフサイクル 🔵

**信頼性**: 🔵 *PRD §2.3より*

一連のフローを通しで検証する E2E テスト:

1. テスト用一時ディレクトリに vault を初期化
2. `DiaryCore::new(vault_path)` で構造体を作成
3. `DiaryCore::unlock(password)` で解錠
4. `new_entry("テスト日記", "本文です", vec!["日記"])` でエントリ作成
5. `list_entries(None)` で 1 件取得を確認
6. `get_entry(id_prefix)` で本文が一致することを確認
7. `update_entry(id, updated_plaintext)` でタイトルを変更
8. `get_entry(id_prefix)` で新しいタイトルを確認
9. `delete_entry(id)` でエントリを削除
10. `list_entries(None)` で 0 件を確認
11. `lock()` で施錠

### ロック中のエントリ操作エラー 🔵

**信頼性**: 🔵 *REQ-092より*

- `DiaryCore::new()` 直後 (unlock 前) に `new_entry()` を呼び出し
- `DiaryError` が返されることを確認
- `lock()` 後に `list_entries()` を呼び出し
- `DiaryError` が返されることを確認

### unlock → lock → unlock サイクル 🔵

**信頼性**: 🔵 *REQ-092より*

- `unlock()` → `new_entry()` (成功) → `lock()` → `new_entry()` (エラー) → `unlock()` → `list_entries()` (成功)

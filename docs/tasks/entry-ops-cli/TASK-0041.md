## 実装詳細

### 1. E2E統合テスト - 完全ワークフロー 🔵

**信頼性**: 🔵 *要件定義書 NFR-001, NFR-002*

Vault初期化からエントリCRUD操作までの完全なワークフローを検証する統合テストを実装する。

**テストシナリオ**:
1. `init` でVaultを初期化
2. `new` でエントリを作成（--password + -b フラグ使用）
3. `list` で作成したエントリが一覧に表示されることを確認
4. `show` でエントリの全文が正しく表示されることを確認
5. `edit` でタイトルとタグを変更
6. `show` で変更が反映されていることを確認
7. `delete --force` でエントリを削除
8. `list` でエントリが一覧から消えていることを確認

**実装ファイル**: `cli/tests/integration_test.rs` または `core/tests/integration_test.rs`

### 2. パスワード優先順位統合テスト 🔵

**信頼性**: 🔵 *要件定義書 REQ-051〜054*

3段階のパスワード取得優先順位が正しく機能することを検証する。

**テストケース**:
- `--password` フラグとenv変数の両方が設定されている場合、フラグが優先される
- env変数のみ設定されている場合、env変数が使用される
- パスワード未指定でTTYでない環境の場合、エラーが返る

**実装ファイル**: `cli/tests/integration_test.rs`

### 3. タグフィルタリング統合テスト 🔵

**信頼性**: 🔵 *要件定義書 REQ-031〜033*

ネストタグの前方一致フィルタリングを検証する。

**テストシナリオ**:
1. タグ `日記`, `日記/旅行`, `技術`, `技術/Rust` のエントリを作成
2. `list --tag "日記"` → `日記` と `日記/旅行` のエントリが表示される
3. `list --tag "日記/旅行"` → `日記/旅行` のエントリのみ表示される
4. `list --tag "技術"` → `技術` と `技術/Rust` のエントリが表示される

**実装ファイル**: `cli/tests/integration_test.rs`

### 4. IDプレフィックス解決統合テスト 🔵

**信頼性**: 🔵 *要件定義書 REQ-041〜043*

IDプレフィックスによるエントリ特定が正しく機能することを検証する。

**テストケース**:
- 4文字のプレフィックスで一意に特定できる場合、エントリが取得される
- 完全なUUIDでエントリが取得される
- プレフィックスが複数エントリに一致する場合の動作を確認

**実装ファイル**: `cli/tests/integration_test.rs`

### 5. エラーケース統合テスト 🔵

**信頼性**: 🔵 *要件定義書 NFR-101*

各種エラーケースが適切にハンドリングされることを検証する。

**テストケース**:
- 不正なパスワードでVaultアンロックが失敗する
- 存在しないIDでエントリ取得が失敗する
- 空のVaultに対する `list` が空の結果を返す

**実装ファイル**: `cli/tests/integration_test.rs`

### 6. doc comments追加 🔵

**信頼性**: 🔵 *Rustドキュメント規約*

Phase 2で実装した全公開型・関数に `///` doc commentsを追加する。

**対象ファイル**:
- `cli/src/password.rs`: `PasswordSource`, `get_password()`, `read_password_tty()`
- `cli/src/editor.rs`: `EditorConfig`, `secure_tmpdir()`, `secure_delete()`, `write_header_file()`, `read_header_file()`, `launch_editor()`
- `cli/src/commands.rs`: `cmd_new()`, `cmd_list()`, `cmd_show()`, `cmd_edit()`, `cmd_delete()`
- `core/src/entry.rs`: 公開型・関数にdoc commentsが不足している箇所

**doc comment規約**:
- 1行目: 関数/型の概要（日本語可）
- 2行目以降: 詳細説明、引数、戻り値、エラーケース
- `# Examples` セクション: 使用例（主要な公開APIのみ）

### 7. sprint-status.md 更新 🔵

- init → new → list → show → edit → show → delete → list の一連のフローが成功することを確認

- 不正パスワード、存在しないID、空Vaultのエラーハンドリングが適切であること

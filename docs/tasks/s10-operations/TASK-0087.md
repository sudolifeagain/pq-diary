### 1. `AppConfig` / `AppSection` 構造体定義

**信頼性**: 🔵 *types.rs L21-39, REQ-601*

`core/src/vault/config.rs` の末尾 (既存 `VaultConfig` 群の後) に追加する:

```rust
/// Application-wide configuration stored in `~/.pq-diary/config.toml`.
#[derive(Debug, PartialEq, Serialize, Deserialize)]
pub struct AppConfig {
    pub app: AppSection,
}

/// `[app]` section of `config.toml`.
#[derive(Debug, PartialEq, Serialize, Deserialize)]
pub struct AppSection {
    pub default_vault: String,
    pub sync_backend: String,
}
```

- `serde` の `Serialize` / `Deserialize` derive を使用 (既存 `VaultConfig` と同じパターン)
- `PartialEq` はテストの assert_eq! のために必要
- フィールド命名は snake_case (TOML のキーと一致)

### 3. `default_path()` / `default_vaults_dir()` 実装

**信頼性**: 🔵 *types.rs L53-70, ヒアリング Q1*

```rust
impl AppConfig {
    pub fn default_path() -> Result<PathBuf, DiaryError> {
        let home = dirs::home_dir()
            .ok_or_else(|| DiaryError::Config("Cannot determine home directory".to_string()))?;
        Ok(home.join(".pq-diary").join("config.toml"))
    }

    pub fn default_vaults_dir() -> Result<PathBuf, DiaryError> {
        let home = dirs::home_dir()
            .ok_or_else(|| DiaryError::Config("Cannot determine home directory".to_string()))?;
        Ok(home.join(".pq-diary").join("vaults"))
    }
}
```

- `dirs::home_dir()` が `None` (ホーム解決失敗) の場合は `DiaryError::Config` を返す
- 戻り値型は `Result<PathBuf, DiaryError>`

**重要 (CLAUDE.md 規約)**: `core/` クレートは現状 `dirs` 依存を持たない。本タスクで以下のいずれかの判断が必要:

- **方針 A**: `core/Cargo.toml` にも `dirs = "5"` を追加 (シンプル、`AppConfig::default_path()` を `core/` 内で実装)
- **方針 B**: `default_path()` / `default_vaults_dir()` を `cli/` 側のヘルパー関数として実装し、`core/AppConfig` は構造体と TOML I/O のみ提供

設計 (`types.rs`) では方針 A を想定 (`core/src/vault/config.rs` に配置)。**本タスクでは方針 A を採用** し、TASK-0086 で追加した `dirs` を `core/Cargo.toml` にも追加する。

### 4. `from_file()` 実装

**信頼性**: 🔵 *REQ-604, types.rs L74-77, 既存 VaultConfig::from_file パターン*

```rust
impl AppConfig {
    pub fn from_file(path: &Path) -> Result<Self, DiaryError> {
        let content = std::fs::read_to_string(path)?;
        toml::from_str(&content).map_err(|e| DiaryError::Config(e.to_string()))
    }
}
```

- `std::io::Error` は `DiaryError::Io` に自動変換 (`#[from]`)
- `toml::de::Error` は `DiaryError::Config` にマップ (EDGE-005)

### 5. `to_file()` 実装 (Unix 0o600 設定)

**信頼性**: 🔵 *REQ-611, types.rs L81-90, 既存 VaultConfig::to_file パターン*

```rust
impl AppConfig {
    pub fn to_file(&self, path: &Path) -> Result<(), DiaryError> {
        let content = toml::to_string(self).map_err(|e| DiaryError::Config(e.to_string()))?;
        std::fs::write(path, content)?;
        #[cfg(unix)]
        {
            use std::os::unix::fs::PermissionsExt as _;
            std::fs::set_permissions(path, std::fs::Permissions::from_mode(0o600))?;
        }
        Ok(())
    }
}
```

- `#[cfg(unix)]` ガードで Unix のみ `0o600` を設定
- Windows ではデフォルト ACL のまま (schema.md L37 に明記)
- `PermissionsExt` の use は `#[cfg(unix)]` ブロック内に閉じる (Windows ビルドの未使用 import 警告を避ける)

### テスト詳細

**TC-S10-087-01**: `AppConfig::default()` を呼び出し、`config.app.default_vault == "default"` および `config.app.sync_backend == "git"` を `assert_eq!` で検証する。

**TC-S10-087-02**: `AppConfig::default()` を作成 → `tempfile::NamedTempFile` のパスへ `to_file` → 同じパスから `from_file` で読み戻し → `assert_eq!` で完全一致を検証する。

**TC-S10-087-03**: `AppConfig::default_path()?` を呼び出し、戻り値の `PathBuf` が `.pq-diary/config.toml` で終わることを `ends_with` で検証する。`dirs::home_dir()` の結果に依存するため、ホーム未設定環境ではスキップ。

**TC-S10-087-04**: 同上の方式で `default_vaults_dir()` が `.pq-diary/vaults` で終わることを検証する。

**TC-S10-087-05** (`#[cfg(unix)]`): `tempfile` で一時パスを作成 → `AppConfig::default().to_file(&path)?` → `std::fs::metadata(&path)?.permissions().mode() & 0o777 == 0o600` を検証する。

**TC-S10-087-06**: `"this is not valid = toml ["` のような不正 TOML 文字列を一時ファイルに書き出し → `AppConfig::from_file(&path)` の戻り値が `Err(DiaryError::Config(_))` であることを `matches!` で検証する。

**TC-S10-087-07**: 存在しないパスを `AppConfig::from_file` に渡し、戻り値が `Err(DiaryError::Io(_))` であることを検証する (NotFound)。

**TC-S10-087-08**: `AppConfig::default()` を `toml::to_string(&config)?` でシリアライズし、出力文字列に `[app]`、`default_vault = "default"`、`sync_backend = "git"` の 3 文字列が全て含まれることを検証する。

## 注意事項

- **CLAUDE.md Rust 規約遵守**:
  - `unsafe` 不使用 (本タスクでは該当なし)
  - 本タスクで扱うデータは秘密ではない (デフォルト vault 名・バックエンド名のみ) ため `zeroize` 不要
  - `unwrap()` / `expect()` はテストコードのみ。本番コードは必ず `Result` を返す
  - エラーは `thiserror` の `DiaryError` を返す。`anyhow` は cli/ 専用のため core/ では使わない
- **プラットフォーム分岐**: Unix の `0o600` 設定は `#[cfg(unix)]` で囲み、`PermissionsExt` の use もブロック内に閉じること (Windows ビルドの未使用 import 警告回避)
- **`dirs` の core/ への追加判断**: 本タスクで採用する方針 A (`core/Cargo.toml` に `dirs = "5"` 追加) は、CLAUDE.md「core/ にプラットフォーム依存 UI コードを入れない」と緊張関係にある。しかし `dirs` は OS 抽象化ライブラリであり「UI コード」ではないため許容範囲と判断する。代替の方針 B (cli/ 側で path 解決) を採用する場合は本タスク開始時に再ヒアリングすること
- **既存 VaultConfig との混在**: `core/src/vault/config.rs` には既に `VaultConfig` 群が存在する。`AppConfig` 群は明確にコメントセクションで区切り、ファイル肥大化を防ぐ。将来 200 行を超えるようなら別モジュール (`core/src/vault/app_config.rs`) への切り出しを検討
- **`tempfile` の dev-dependencies**: 既存テストで使用されている想定だが、未追加の場合は `core/Cargo.toml` の `[dev-dependencies]` に追加すること
- **DiaryError バリアント追加なし**: 本タスクでは既存の `Config` / `Io` バリアントのみ使用するため、`DiaryError` 自体の変更は不要 (types.rs L280-300 で確認済み)
- **doc comments**: 全公開 API (構造体 / 関数) に英語 `///` コメントを付与。`# Arguments` / `# Returns` / `# Errors` セクションを主要 API には記載 (S9 doc comments 規約踏襲)

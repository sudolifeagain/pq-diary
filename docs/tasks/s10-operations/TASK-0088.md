## タスク概要

`pq-diary info --security` コマンドから実プロセスのセキュリティ強化状態を取得するため、`cli/src/security.rs` に `HardenStatus` 構造体と `harden_status()` (HardenStatus::current()) 関数を追加する。Phase 2 の `cmd_info` 実装で表示用に利用される基盤 API。

設計は `types.rs` L99-139 で完全に確定済み。

### 1. `HardenStatus` 構造体定義

**信頼性**: 🔵 *types.rs L99-113, REQ-411, REQ-412*

`cli/src/security.rs` の末尾 (既存テスト `mod tests` の前) に追加する:

```rust
/// Snapshot of process hardening state for `info --security`.
///
/// Each field reflects live process state (NFR-104 forbids hardcoded values).
#[derive(Debug, Clone, Copy, PartialEq, Eq)]
pub struct HardenStatus {
    /// `mlock` / `VirtualLock` is currently locking secret pages.
    pub mlock_active: bool,
    /// Core dumps are disabled.
    ///
    /// - Unix: `Dumpable == 0` AND `RLIMIT_CORE == (0, 0)`
    /// - Windows: always `true` (no coredump concept)
    pub coredump_disabled: bool,
    /// A debugger is currently attached.
    ///
    /// - Unix: `/proc/self/status` `TracerPid` != 0
    /// - Windows: `IsDebuggerPresent()` != 0
    pub debugger_detected: bool,
}
```

- `Debug` / `Clone` / `Copy` / `PartialEq` / `Eq` derive (types.rs L105 と一致)
- 3 フィールドはすべて `bool`

### 2. `HardenStatus::current()` 実装

impl HardenStatus {
    /// Probe the current process for hardening state.
    ///
    /// Returns a snapshot reflecting actual process state, not hardcoded values

            coredump_disabled: query_coredump_disabled(),
            debugger_detected: query_debugger_detected(),

    }
}

### 3. `query_coredump_disabled()` 実装 (Unix)

**信頼性**: 🔵 *REQ-412, types.rs L120-122, S9 既存 harden_process ロジック*

```rust
#[cfg(unix)]
fn query_coredump_disabled() -> bool {
    // Check 1: /proc/self/status Dumpable == 0
    let dumpable_zero = match std::fs::read_to_string("/proc/self/status") {
        Ok(status) => status.lines().any(|line| {
            line.starts_with("Dumpable:")
                && line
                    .split(':')
                    .nth(1)
                    .unwrap_or("1")
                    .trim()
                    .parse::<u32>()
                    .unwrap_or(1)
                    == 0
        }),
        Err(_) => false, // /proc unavailable (e.g. macOS) → cannot confirm
    };

    // Check 2: RLIMIT_CORE soft and hard both 0
    use nix::sys::resource::{getrlimit, Resource};
    let rlimit_zero = matches!(getrlimit(Resource::RLIMIT_CORE), Ok((0, 0)));

    dumpable_zero && rlimit_zero
}

#[cfg(windows)]
fn query_coredump_disabled() -> bool {
    // Windows has no coredump concept; treat as always-disabled.
    true
}
```

- macOS 等 `/proc` が無い環境では `dumpable_zero=false` とし、保守的に `false` を返す (NFR-104: 正確性優先)
- `unwrap_or` は CLAUDE.md ルール上許容 (S9 既存パターンと同じ)
- S9 で既に `nix::sys::resource::{getrlimit, Resource}` を import しているため依存追加不要

### 4. `query_debugger_detected()` 実装

**信頼性**: 🔵 *REQ-412, types.rs L123, S9 既存 check_debugger ロジック*

```rust
#[cfg(unix)]
fn query_debugger_detected() -> bool {
    match std::fs::read_to_string("/proc/self/status") {
        Ok(status) => status.lines().any(|line| {
            line.starts_with("TracerPid:")
                && line
                    .split(':')
                    .nth(1)
                    .unwrap_or("0")
                    .trim()
                    .parse::<u32>()
                    .unwrap_or(0)
                    != 0
        }),
        Err(_) => false, // /proc unavailable → assume no debugger
    }
}

#[cfg(windows)]
fn query_debugger_detected() -> bool {
    // SAFETY: IsDebuggerPresent() is a nullary Win32 API, always safe to call.
    // Listed in CLAUDE.md allowed unsafe scope.
    let attached =
        unsafe { windows_sys::Win32::System::Diagnostics::Debug::IsDebuggerPresent() };
    attached != 0
}

#[cfg(not(any(unix, windows)))]
fn query_debugger_detected() -> bool {
    false
}
```

- S9 既存 `check_debugger()` は `eprintln!` で warning を出す副作用付き。本タスクの `query_debugger_detected()` は副作用なしで bool を返す純粋関数
- `unsafe { IsDebuggerPresent() }` は S9 既存実装と同一の SAFETY コメント付き
- macOS など `cfg(not(any(unix, windows)))` ではフォールバック `false`

### 5. `query_mlock_active()` 実装

**信頼性**: 🔵 *types.rs L119, L130 + S9 secure_mem.rs*

S9 実装の `secure_mem.rs` が `is_locked()` 等の状態取得 API を提供しているかを確認し、以下のいずれかを採用:

- **方針 A (推奨)**: `secure_mem::is_locked()` が存在すれば呼び出す
- **方針 B (フォールバック)**: `secure_mem` 内の静的フラグ (例: `MLOCK_ATTEMPTED` 等) を参照
- **方針 C (簡略化)**: 「`harden_process()` 系の mlock 呼び出しが 1 回でも成功したか」を本ファイル内の `AtomicBool` で追跡

実装着手時に `core/src/secure_mem.rs` (S9 で実装) の API を確認し、最も自然な方法を採用する。

```rust
fn query_mlock_active() -> bool {
    // Implementation depends on S9 secure_mem API surface.
    // - If `core::secure_mem::is_locked() -> bool` exists, delegate.
    // - Otherwise, track an AtomicBool set by harden_process().
    core::secure_mem::is_locked()  // 方針 A の例 (要確認)
}
```

**実装方針の確定タイミング**: tdd-requirements 段階で `secure_mem.rs` を読み、上記 A/B/C のどれを採用するか決定し、設計レビュー記録を `docs/implements/s10-operations/TASK-0088/` に残す。

## 注意事項

- **CLAUDE.md Rust 規約遵守**:
  - `unsafe` は `IsDebuggerPresent` のみ使用 (CLAUDE.md 許可リスト内)。S9 既存と同じ SAFETY コメントを付与
  - 秘密データを扱わないため `zeroize` は不要 (`HardenStatus` の 3 フィールドは bool のみ)
  - `unwrap()` / `expect()` はテストコードのみ。本番コードは `unwrap_or` で安全なフォールバック値を返す (S9 既存パターン踏襲)
  - 本番コードは bool を返す純粋関数。`Result` ではなく `bool` を返す設計は types.rs L127-133 で確定済み (UI 表示用のため失敗時は安全側の値で十分)
- **プラットフォーム分岐**: `#[cfg(unix)]` / `#[cfg(windows)]` / `#[cfg(not(any(unix, windows)))]` の 3 つで全プラットフォームをカバー。`windows_sys::Win32::System::Diagnostics::Debug::IsDebuggerPresent` の use は `#[cfg(windows)]` 関数内のみで使用
- **既存 API 不変条件**: `harden_process()` / `check_debugger()` のシグネチャ・副作用・doc comments を変更しないこと。S9 テスト TC-S9-084-01 〜 05 が全通過することが完了条件
- **`mlock_active` の取得方針**: S9 `secure_mem.rs` の API surface を実装着手時に確認し、A (既存 `is_locked()` 流用) / B (静的フラグ参照) / C (本ファイル内 `AtomicBool`) のどれを選んだかを `docs/implements/s10-operations/TASK-0088/design-note.md` に記録すること。Phase 2 で `cmd_info` 実装者が判断材料として参照する
- **NFR-104 (実プロセス状態反映) 厳守**: 「常に true / 常に false」を返さない。Windows の `coredump_disabled = true` は概念不在 (= 該当事象が発生しえない) のため例外的に許容。types.rs L130-131 で明記済み
- **macOS 等 `/proc` 不在環境のハンドリング**: `coredump_disabled` は保守的に `false` を返す (確認不能なので「無効化されているとは断定できない」)。`debugger_detected` は `false` を返す (デバッガ検知できないので「検知されない」)。両者の方針が逆である理由を doc comment に明記
- **DiaryError バリアント追加なし**: 本タスクは bool 返却のみで Result を返さないため、`DiaryError` 変更は不要
- **`AtomicBool` 採用時のスレッドセーフ性**: 方針 C を採用する場合、`Ordering::SeqCst` で読み書きすること
- **doc comments**: 全公開 API (`HardenStatus`, `current()`, 各フィールド) に英語 `///` を付与。`# Examples` を `current()` に追加 (`info --security` 想定の最小利用例)

## タスク概要

Phase 1 で `not_implemented("init", "Sprint 2")` を返していた `pq-diary init` を本実装する。`~/.pq-diary/config.toml` (AppConfig) と `~/.pq-diary/vaults/default/` (default vault, policy=none) を一発初期化する。NFR-201「初回ユーザーが引数なし `pq-diary init` のみで使用開始できる」を実現する。vault 作成失敗時は config.toml と部分ディレクトリを zeroize 削除し、init が冪等になるよう "Already initialized" で安全側に倒す。

## 実装詳細

### 1. dispatch 切り替え (cli/src/main.rs)

**信頼性**: 🔵 *cli-commands.md §1 dispatch 変更*

`Commands::Init` のディスパッチを置換する:

```rust
- Commands::Init => not_implemented("init", "Sprint 2"),
+ Commands::Init => commands::cmd_init(cli),
```

- 既存 `not_implemented("init", "Sprint 2")` の呼び出しは完全に消去する (S10 REQ-703 で「残った not_implemented」は "Planned for Phase 2" に統一する整理対象だが、init は本実装するため呼び出し自体を削除)
- `cli` (`&Cli`) を渡すのは `--password` / `--claude` / `--debug` 等のグローバルフラグを `cmd_init` 内で参照可能にするため

### 2. `cmd_init(cli: &Cli) -> anyhow::Result<()>` 実装 (cli/src/commands.rs)

**信頼性**: 🔵 *REQ-101~112 + dataflow.md init シーケンス*

実装順序:

1. **config.toml パス解決**: `AppConfig::default_path()` を呼び `~/.pq-diary/config.toml` の `PathBuf` を得る (`DiaryError::Io` を anyhow に伝搬)
2. **重複チェック (REQ-111 / EDGE-104)**: `config_path.exists()` で true なら `anyhow::bail!("Already initialized at {}", parent_dir.display())` で停止 — 既存データには手を触れない
3. **パスワード取得 (REQ-102, NFR-201)**: 既存 `get_password(cli, "New password: ")` を利用。優先順位は `--password` > `PQ_DIARY_PASSWORD` > TTY (S1 既存パターン)。TTY 入力は `rpassword` 等で echo オフ
4. **空パスワードガード (REQ-312 と整合)**: `SecretString::expose_secret().is_empty()` チェックで `anyhow::bail!("New password must not be empty")` (TC-101-E02)
5. **`~/.pq-diary/` 作成 (EDGE-001)**: `fs::create_dir_all(parent_dir)?`。失敗時 `anyhow::Context::context("Cannot create config directory")` で `DiaryError::Io` を包む
6. **AppConfig 書き出し**: `AppConfig::default().to_file(&config_path)?` — TASK-0087 の API。Unix では内部で `set_permissions(0o600)` (REQ-611)
7. **VaultManager で default vault 作成**: `VaultManager::new(parent_dir.join("vaults"))?.create_vault("default", &pwd, Policy::None)?`
8. **vault 作成失敗時クリーンアップ (REQ-112, EDGE-002)**: ステップ 7 が `Err` の場合:
   - `fs::remove_file(&config_path)` で config.toml 削除
   - `fs::remove_dir_all(parent_dir.join("vaults"))` で部分 vault ディレクトリ削除 (存在すれば)
   - 削除はベストエフォート (削除失敗はログのみ、元エラーを優先返却)
   - クリーンアップ後 `anyhow::bail!("Vault creation failed: {}", err)` を返す
9. **成功メッセージ (REQ-104)**: `println!("Initialized pq-diary at {}", parent_dir.display())` を出力し `Ok(())`

### 3. メモリ保護

**信頼性**: 🔵 *CLAUDE.md Rust 規約 + NFR-101*

- パスワードは `SecretString` で受け取り、`cmd_init` スコープ離脱時に Drop で zeroize される
- 生の `String` / `Vec<u8>` でパスワードを保持しない (CLAUDE.md 規約)
- `unwrap()` / `expect()` は使わず全て `?` または明示的なエラー伝搬

### 4. NFR-001 (init < 3 秒) 対応

**信頼性**: 🔵 *NFR-001 + TC-101-B01*

- 本番パラメータ (Argon2id memory_cost_kb=65536, time_cost=3, parallelism=1) で 3 秒以内が要件
- テストでは S9 TASK-0085 の `fast_params()` パターンに従い `Argon2Params::fast_params()` で計測 (タイミング不安定回避)
- 本番パラメータでの 3 秒測定は手動検証 (CI では fast_params)

### テスト詳細

**TC-101-01** 🔵: `tempfile::TempDir` で一時 HOME を作成、`std::env::set_var("HOME", tmp)` (Unix) or `USERPROFILE` (Windows) し、`PQ_DIARY_PASSWORD="Test123!"` をセットして `cmd_init(&cli)` を呼ぶ。完了後 `{tmp}/.pq-diary/config.toml`, `{tmp}/.pq-diary/vaults/default/vault.pqd`, `vault.toml`, `entries/` の存在を `Path::exists()` で検証。`AppConfig::from_file` で読み戻し `default_vault == "default"` / `sync_backend == "git"` を確認。VaultConfig を読み policy が `Policy::None` であることを確認。

**TC-101-02** 🔵: TC-101-01 と同じセットアップ。`--password` フラグなし、TTY なし環境 (テストプロセスは isatty=false) で `PQ_DIARY_PASSWORD` 経由のパスワード取得経路をカバー。get_password の優先順位 (env > TTY) を検証。

**TC-101-E01** 🔵: 事前に `{tmp}/.pq-diary/config.toml` をダミー内容で作成。`cmd_init` を呼んで `Err` 返却、エラーメッセージに "Already initialized" を含むことを `format!("{:#}", err).contains("Already initialized")` で検証。既存 config.toml のバイト列が無変更であることを確認 (mtime と内容のハッシュ比較)。

**TC-101-E02** 🔵: `PQ_DIARY_PASSWORD=""` で `cmd_init` を呼ぶ。`Err` 返却 + "must not be empty" メッセージ。`~/.pq-diary/config.toml` および `vaults/` が一切作成されていないことを `!path.exists()` で検証。

**TC-101-E04** 🔵: VaultManager の `create_vault` が write 中に失敗するシナリオを再現する。実装方針: `~/.pq-diary/vaults/` を事前に読み取り専用ファイル (同名ファイル) で塞ぐか、`#[cfg(unix)]` で `chmod 0o500` した親ディレクトリにする。`cmd_init` が `Err` を返し、その後 `{tmp}/.pq-diary/config.toml` および `{tmp}/.pq-diary/vaults/default/` が削除されている (`!exists()`) ことを検証。

**TC-101-B01** 🔵: `Argon2Params::fast_params()` を使用するテストモードで `cmd_init` を呼び、`std::time::Instant::now()` で経過時間を計測。本番閾値 < 3000ms を fast_params 環境では十分余裕で達成 (S9 TASK-0085 と同じパターン)。本番パラメータでの 3 秒テストは `#[ignore]` で別途用意し手動実行可能にする。

## 注意事項 (CLAUDE.md Rust 規約)

- `unsafe` は使用しない (init コマンドは mlock/VirtualLock/PR_SET_DUMPABLE 等のプラットフォーム API を直接呼ばない。S9 TASK-0084 で実装済みの `harden_process()` は main.rs で呼ばれるためここでは不要)
- パスワードは必ず `SecretString` で受け取り、生の `String` で保持しない
- `unwrap()` / `expect()` はテストコードのみ許可。本番コードは全て `Result` 返却 + `?` 演算子
- エラーは `cli/` レイヤなので `anyhow` を許可 (`anyhow::bail!`, `anyhow::Context`)。`core/` から伝搬する `DiaryError` は `?` で anyhow に変換
- プラットフォーム分岐 (`HOME` vs `USERPROFILE`) は `dirs::home_dir()` で吸収し `#[cfg]` を最小化
- vault 作成失敗時のクリーンアップ削除は失敗しても元のエラーを優先返却 (二重 Err にしない)
- doc comment (`///`) を `cmd_init` の公開 API には英語で付与する (S9 TASK-0085 と同じドキュメント規約)

## タスク概要

Phase 1 で `not_implemented("sync", "Sprint 8")` を返していた `pq-diary sync` を本実装する。AppConfig (`~/.pq-diary/config.toml`) の `sync_backend` フィールドを読み、現状唯一サポートする `"git"` バックエンドであれば既存の `cmd_git_sync(cli)` (S8 実装済) に委譲する config 駆動ディスパッチャを実装する。AppConfig 不在時は init 誘導エラー、未知のバックエンド値は即停止する。S8 で実装した `git-sync` を AppConfig 経由で抽象化することで、将来 (Phase 3+) の他バックエンド追加 (rsync 等) に拡張可能な土台を作る。

## 実装詳細

### 1. dispatch 切り替え (cli/src/main.rs)

**信頼性**: 🔵 *cli-commands.md §2 dispatch 変更*

`Commands::Sync` のディスパッチを置換する:

```rust
- Commands::Sync => not_implemented("sync", "Sprint 8"),
+ Commands::Sync => commands::cmd_sync(cli),
```

- 既存 `not_implemented("sync", "Sprint 8")` の呼び出しは完全に消去する (REQ-703 の "Planned for Phase 2" 統一は他の未実装コマンドのみ対象。sync は本実装するため呼び出しごと削除)
- `cmd_git_sync` は S8 で実装済の `commands::cmd_git_sync(cli: &Cli)` を流用 (再エクスポート不要、commands モジュール内で直接呼ぶ)

### 2. `cmd_sync(cli: &Cli) -> anyhow::Result<()>` 実装 (cli/src/commands.rs)

**信頼性**: 🔵 *REQ-201~212 + dataflow.md sync シーケンス*

実装順序:

1. **config.toml パス解決**: `AppConfig::default_path()?` で `~/.pq-diary/config.toml` の `PathBuf` を取得
2. **AppConfig 不在チェック (REQ-211, EDGE-005)**: `config_path.exists()` が false の場合 `anyhow::bail!("pq-diary init を先に実行してください (~/.pq-diary/config.toml が見つかりません)")` で停止 — `init` 誘導メッセージは日本語固定 (ヒアリング2026-05-17 確定提案H)
3. **AppConfig 読み込み (REQ-201)**: `AppConfig::from_file(&config_path)?` を呼び出す。TOML パースエラーは `DiaryError::Config("Invalid config.toml: {error}")` (TASK-0087 内で実装) として伝搬され、`?` で anyhow に変換される
4. **`sync_backend` 判定 (REQ-202, REQ-212)**:
   - `config.app.sync_backend.as_str()` で値取得
   - **デフォルト処理 (REQ-212)**: `AppConfig::Default` が `sync_backend = "git"` を返すため、TOML 内でフィールドが省略されていても serde の `#[serde(default)]` で `"git"` が入る (TASK-0087 で実装)。`cmd_sync` 側で追加のデフォルト処理は不要
   - **"git" 分岐 (REQ-202)**: `cmd_git_sync(cli)` を呼び出し、その `Result` をそのまま返却
   - **未知値 (REQ-203)**: `anyhow::bail!("Unknown sync backend: {}", value)` で停止

### 3. エラーハンドリング

**信頼性**: 🔵 *EDGE-005 + dataflow.md エラーハンドリングフロー*

| ケース | 動作 |
|---|---|
| `config.toml` 不在 | `anyhow::bail!("pq-diary init を先に実行してください (~/.pq-diary/config.toml が見つかりません)")` |
| `config.toml` TOML 破損 | `AppConfig::from_file` が `DiaryError::Config("Invalid config.toml: {detail}")` を返却 → `?` で anyhow 伝搬 |
| `sync_backend = "git"` | `cmd_git_sync(cli)` の結果を返却 (pull→push) |
| `sync_backend` 省略 | serde default で `"git"` 扱い (cmd_git_sync 呼び出し) |
| `sync_backend = "github"` 等の未知値 | `anyhow::bail!("Unknown sync backend: {}", value)` |

### 4. メモリ保護

**信頼性**: 🔵 *CLAUDE.md Rust 規約*

- sync コマンド自体はパスワード入力を行わない (内部で呼ぶ `cmd_git_sync` がパスワードを取得する)
- AppConfig は秘密情報を含まないため通常の `String` で OK
- 生の `unwrap()` / `expect()` は本番コードで使用しない

### テスト詳細

**TC-201-01** 🔵: `tempfile::TempDir` で一時 HOME を作成、`AppConfig { default_vault: "default", sync_backend: "git" }` を `config.toml` として書き出し、vault 側を git-init 済の状態でセットアップ。`cmd_sync(&cli)` を呼ぶ。検証方針: `cmd_git_sync` を呼んだことを確認するため、git ローカル bare リポジトリをテスト内でセットアップし pull→push が走ることを副作用 (リポジトリの状態変化) で検証する。`cmd_git_sync` 内部のモック化は避け、S9 TASK-0085 の git ライフサイクル E2E パターン (実 git CLI + 一時 bare repo) に従う。

**TC-201-02** 🔵: AppConfig を `default_vault = "default"` のみで書き出し (`sync_backend` フィールドを省略)。`AppConfig::from_file` で serde default が効き `sync_backend = "git"` になることを `assert_eq!` で確認した上で、`cmd_sync` が TC-201-01 と同じ動作 (cmd_git_sync 呼び出し) をすることを検証。

**TC-201-E01** 🔵: AppConfig を `sync_backend = "github"` で書き出し、`cmd_sync` 呼び出しが `Err` 返却、エラーメッセージに "Unknown sync backend: github" を含むことを `format!("{:#}", err).contains("Unknown sync backend: github")` で検証。vault の状態 (git index, working tree) が無変更であることも確認 (副作用なし)。

**TC-201-E02** 🔵: 一時 HOME のみ作成し、`~/.pq-diary/config.toml` を作成せずに `cmd_sync` を呼ぶ。`Err` 返却 + エラーメッセージに "pq-diary init を先に実行してください" を含むことを検証。`~/.pq-diary/` および `vaults/` が一切作成されないことも確認 (sync は副作用ゼロ)。

**TC-201-E03** 🔵: `~/.pq-diary/config.toml` に不正な TOML (例: `default_vault = [unclosed`) を書き出してから `cmd_sync` を呼ぶ。`Err` 返却 + エラーメッセージに "Invalid config.toml" を含むことを検証。`DiaryError::Config` バリアントが anyhow 経由で伝搬されている (Source chain で確認可能) ことを検証。

## 注意事項 (CLAUDE.md Rust 規約)

- `unsafe` は使用しない (sync コマンドはプラットフォーム API を直接呼ばない)
- パスワード関連は `cmd_git_sync` 側に委譲、`cmd_sync` 内では `SecretString` を扱わない
- `unwrap()` / `expect()` はテストコードのみ許可。本番コードは全て `Result` 返却 + `?` 演算子
- エラーは `cli/` レイヤなので `anyhow` を許可 (`anyhow::bail!`, `anyhow::Context`)。`core/` から伝搬する `DiaryError::Config` / `DiaryError::Io` は `?` で anyhow に変換
- AppConfig は秘密情報を含まないため `zeroize` は不要 (vault.toml の Argon2 パラメータも同様)
- プラットフォーム分岐は `AppConfig::default_path()` (TASK-0087) 内で `dirs::home_dir()` により吸収済。`cmd_sync` 側に `#[cfg]` を入れない
- doc comment (`///`) を `cmd_sync` の公開 API には英語で付与する (S9 TASK-0085 と同じドキュメント規約)
- Phase 3 以降のバックエンド追加 (rsync, github CLI 等) を想定し、`sync_backend` 値の match は明示的に書く (`_ => bail!(...)`) — 将来の追加バリアントを忘れないため

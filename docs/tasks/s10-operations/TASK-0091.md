## タスク概要

`change-password` の中核となる再暗号化処理 `re_encrypt_vault` をコアロジックとして実装する。旧パスワードで vault を unlock → 全エントリを `SecretBytes` で取得 → 新パスワードで `vault.pqd.tmp` に再書き出し → `rename` でアトミック差し替えを行う。CLI 層から独立した純粋なコアロジックとしてテスト可能な形で実装し、TASK-0092 (CLI 実装) から呼び出す。

vault.pqd の更新範囲は `kdf_salt` 再生成、`verification_iv/ct`、`kem_encrypted_sk`、`dsa_encrypted_sk`、エントリペイロードに限定し、KEM/DSA の seed そのものは保持する (公開鍵不変)。失敗時は `vault.pqd.tmp` を zeroize 削除して旧 `vault.pqd` を維持する。メモリ管理は `ChangePasswordContext` 構造体に集約し、Drop で旧鍵・新鍵・全エントリを zeroize する。

## 実装詳細

### 1. `re_encrypt_vault` 関数シグネチャ

**信頼性**: 🔵 *types.rs `re_encrypt_vault` シグネチャ + design dataflow*

```rust
pub fn re_encrypt_vault(
    vault_dir: &Path,
    old_pwd: &SecretString,
    new_pwd: &SecretString,
) -> Result<(), DiaryError>
```

- 配置先: `core/src/` (例: `core/src/vault/change_password.rs` 新規) または `cli/src/commands.rs` のヘルパー (設計判断は TDD-red フェーズで確定)。
- 戻り値: `Result<(), DiaryError>` (core API は `anyhow` 不使用、CLAUDE.md 規約)。
- 引数の `old_pwd` / `new_pwd` は呼び出し側が `SecretString` で保持し、関数内では参照のみ扱う。

### 2. 処理フロー

**信頼性**: 🔵 *dataflow.md 機能 3 シーケンス + schema.md change-password 更新範囲表*

1. **旧パスワードで unlock**: `DiaryCore::new(vault_dir)` → `unlock(old_pwd)`
   - 失敗時: `DiaryError::Crypto` をそのまま返す (REQ-311 のメッセージは CLI 側で整形)
2. **全エントリ取得**: `list_entries_with_body(&core)` で `Vec<EntryPlaintext>` を取得
   - `EntryPlaintext` は `ZeroizeOnDrop` 実装済 (S5)、`body` は `SecretBytes`
3. **新パスワードで tmp vault 初期化**:
   - `vault.pqd.tmp` ファイル名で `VaultManager::init_vault(tmp_path, new_pwd, Policy::None)` 相当の処理
   - kdf_salt は新規ランダム生成
   - KEM/DSA seed は旧 vault から復号して保持し、新鍵で再暗号化
4. **全エントリを tmp に書き込み**:
   - 各 `EntryPlaintext` を新鍵で AES-256-GCM 暗号化
   - ML-DSA-65 で署名 (seed は不変なので署名鍵も同一)
   - HMAC-SHA256 を新鍵から派生したサブキーで計算
5. **成功時**: `fs::rename(vault_pqd_tmp, vault_pqd)` でアトミック差し替え
6. **失敗時**: `vault.pqd.tmp` を zeroize して削除、旧 `vault.pqd` 維持
7. **Drop**: `ChangePasswordContext` のスコープ離脱時に旧鍵・新鍵・全エントリ・KEM/DSA seed が zeroize される

### 3. vault.pqd 更新範囲 🔵

| フィールド | 更新? | 備考 |
|---|---|---|
| schema_version | NO | v4 維持 |
| flags | NO | 予約 |
| payload_size | YES (同値) | 再計算 |
| **kdf_salt** | **YES** | 新パスワード用ランダム再生成 |
| legacy_salt | NO | デジタル遺言用、別マスター鍵で派生 |
| **verification_iv / verification_ct** | **YES** | 新鍵で再暗号化 |
| kem_pk_offset | NO | 予約 |
| dsa_pk_hash | NO | 公開鍵不変 |
| **kem_encrypted_sk** | **YES** | 同じ KEM seed を新鍵で再暗号化 |
| **dsa_encrypted_sk** | **YES** | 同じ DSA seed を新鍵で再暗号化 |
| エントリペイロード | YES | 全エントリを新鍵で再暗号化 |

**重要**: KEM/DSA seed そのものは変更しない。これにより過去エントリの署名検証が引き続き成功する (`dsa_pk_hash` 不変)。

### 4. ChangePasswordContext 構造体

**信頼性**: 🔵 *NFR-101 + CLAUDE.md Rust 規約 (zeroize)*

```rust
struct ChangePasswordContext {
    old_key: ZeroizingKey,
    new_key: ZeroizingKey,
    kem_seed: SecretBytes,
    dsa_seed: SecretBytes,
    entries: Vec<EntryPlaintext>,  // 各要素は ZeroizeOnDrop
}

impl Drop for ChangePasswordContext {
    fn drop(&mut self) {
        // 各フィールドは Zeroize/ZeroizeOnDrop を実装済のため、
        // フィールドの Drop が自動的に呼ばれる。
        // ここで明示的に zeroize を呼ぶ必要はないが、
        // 漏れがあれば self.entries.clear() などで明示する。
    }
}
```

- 旧鍵・新鍵は `ZeroizingKey` (S2 で実装) でラップ
- KEM/DSA seed は `SecretBytes` でラップ
- エントリは `Vec<EntryPlaintext>` (`EntryPlaintext` は S5 で `ZeroizeOnDrop` 実装済)
- 構造体全体が Drop されるとフィールドの Drop が連鎖し、全てのデータが zeroize される

### 5. アトミック書き込みパターン

**信頼性**: 🔵 *既存 core/src/vault/writer.rs `.tmp + rename` パターン*

- `.tmp` ファイル名: `vault.pqd.tmp` (writer.rs の既存パターン踏襲)
- 成功時のみ `fs::rename(tmp_path, final_path)` でアトミック差し替え
- 失敗時の zeroize 削除: `fs::OpenOptions::write(true).open(tmp_path)` で開いて 0 で上書き → `fs::remove_file` (ベストエフォート)

**TC-CP-CORE-01**: テスト用空 vault を旧パスワード "Old123!" で作成し、`re_encrypt_vault` を新パスワード "New456!" で呼び出す。完了後、旧パスワードで `DiaryCore::unlock` が `DiaryError::Crypto` を返し、新パスワードで `Ok` を返すことを検証する。

**TC-CP-CORE-02**: 3 エントリを含む vault を作成し、各エントリの (title, tags, body) を記録する。`re_encrypt_vault` 後、新パスワードで unlock して `list_entries_with_body` で全エントリを取得し、(title, tags, body) が完全一致することを検証する。

**TC-CP-CORE-03**: 変更前後で vault.pqd の `dsa_pk_hash` を読み取り、同一であることを検証する。過去エントリの ML-DSA-65 署名が変更後も検証成功することも確認する。

**TC-CP-CORE-04**: 変更前後で vault.pqd の `kdf_salt` を読み取り、異なる値であることを検証する (旧パスワードからは新鍵を派生不可能であることの担保)。

**TC-CP-CORE-06**: 100 エントリ (各 1KB) を持つ vault を作成し、`std::time::Instant` で `re_encrypt_vault` の実行時間を計測。30 秒以内 (fast_params 使用) であることを検証する。

**TC-CP-CORE-07**: `ChangePasswordContext` をスコープ内で作成し、`drop(context)` 後に内部バッファ (旧鍵・新鍵) が zero フィルされていることを検証する。検証は可能な範囲で (`unsafe` でメモリ位置を保存して確認する手法、または zeroize クレートのテスト機能を利用)。

## 注意事項

- `re_encrypt_vault` は core API として実装し、CLAUDE.md 規約に従い `anyhow` を使用しない (`DiaryError` を返す)
- CLI 層 (TASK-0092) が `DiaryError` を `anyhow::Result` に変換する
- KEM/DSA seed そのものは変更しないこと (schema.md の重要注記)。seed を変えると公開鍵が変わり過去署名の検証が壊れる
- `vault.pqd.tmp` のファイル名は writer.rs の既存パターンと一貫させること
- `EntryPlaintext` は S5 で `ZeroizeOnDrop` 実装済のため、`Vec<EntryPlaintext>` の Drop で各要素が自動 zeroize される
- パフォーマンステストは fast_params() を使用し、S8 で発生したタイミング不安定問題を回避すること (S9 TASK-0085 パターン)
- Drop 検証 (TC-CP-CORE-07) は実装難度が高い場合、zeroize クレートの単体テストへのリンクで代替可能
- 本タスクは TASK-0092 の前提タスクであり、CLI dispatch の切り替えは TASK-0092 で実施する

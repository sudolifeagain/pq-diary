## 実装詳細

### 1. `cmd_change_password` 関数

**信頼性**: 🔵 *cli-commands.md サブコマンド 3 + dataflow.md 機能 3 シーケンス*

```rust
pub fn cmd_change_password(cli: &Cli) -> anyhow::Result<()> {
    // 1. --claude ブロック
    check_claude_policy(cli, "change-password")?;

    // 2. 旧パスワード TTY 取得
    let old_pwd = prompt_password("Old password:")?;

    // 3. unlock で旧パスワード検証
    let vault_dir = resolve_vault_dir(cli)?;
    {
        let mut core = DiaryCore::new(&vault_dir)?;
        core.unlock(&old_pwd).map_err(|_| {
            anyhow::anyhow!("Old password is incorrect")
        })?;
        // core はここで Drop される (lock 解除)
    }

    // 4. 新パスワード TTY × 2 取得
    let new_pwd1 = prompt_password("New password:")?;
    let new_pwd2 = prompt_password("Confirm new password:")?;

    // 5. 不一致チェック
    if new_pwd1.expose_secret() != new_pwd2.expose_secret() {
        anyhow::bail!("Passwords do not match");
    }

    // 6. 空チェック
    if new_pwd1.expose_secret().is_empty() {
        anyhow::bail!("New password must not be empty");
    }

    // 7. 新旧同一警告 (処理は続行)
    if old_pwd.expose_secret() == new_pwd1.expose_secret() {
        eprintln!("Warning: New password is identical to old password.");
    }

    // 8. 再暗号化実行
    re_encrypt_vault(&vault_dir, &old_pwd, &new_pwd1)?;

    // 9. 成功メッセージ
    println!("Password changed successfully");
    Ok(())
}
```

### 3. --claude ブロックの実装

**信頼性**: 🔵 *NFR-103, cli-commands.md*

既存の `check_claude_policy` または同等の判定関数を利用する。`cmd_change_password` の冒頭で呼び出し、`cli.claude == true` の場合は即座に `anyhow::bail!("change-password is not permitted with --claude")` で停止する。

### 4. パスワード入力の UX

**信頼性**: 🔵 *PRD §4.2 + 既存 prompt_password パターン*

- 既存の `prompt_password()` ヘルパー (S9 で `SecretString` 化済) を流用
- 旧パスワード: `"Old password:"` プロンプト
- 新パスワード 1: `"New password:"` プロンプト
- 新パスワード 2: `"Confirm new password:"` プロンプト (REQ-NFR-203)
- 全プロンプトでエコー無効化 (既存 TTY 実装による)
- `--password` フラグや `PQ_DIARY_PASSWORD` 環境変数は **旧パスワードにのみ** 適用 (新パスワードは常に TTY 取得を強制) — 安全側設計

**注意**: `--password` フラグ / 環境変数の扱いについては TDD-red フェーズで再確認すること。仕様上は「旧パスワードは取得優先順位 (flag→env→TTY) に従い、新パスワードは常に TTY × 2」とするのが安全。

### テスト詳細

**TC-301-01**: 空 vault を旧パス "Old123!" で作成し、`cmd_change_password` を新パス "New456!" で実行。完了後、旧パスで `unlock` 失敗、新パスで成功することを検証する。

**TC-301-02**: 3 エントリ vault で change-password 後、新パスで unlock して `list_entries_with_body` で全エントリを取得し、各エントリの (title, tags, body) が変更前と一致することを検証する。

**TC-301-03**: change-password 後に CLI `list` 相当の関数を呼び出し、出力中のタイトル・タグ文字列が変更前と一致することを検証する。

**TC-301-04**: change-password 後に CLI `show <id>` 相当の関数で本文を取得し、変更前のバイト列と完全一致することを検証する。

**TC-301-05**: 100 エントリ vault で `cmd_change_password` の実行時間を `std::time::Instant` で計測し、30 秒以内 (fast_params 使用) であることを検証する。

**TC-301-E01**: 旧パス "Wrong" を入力して `cmd_change_password` を実行し、`Error: Old password is incorrect` が stderr に出力され、vault.pqd の mtime と内容が変更前と一致することを検証する。

**TC-301-E02**: 新パス空文字を 2 回入力 (実装上は新パス 1 を入力後に空チェック)。`Error: New password must not be empty` が出力され vault 無変更を検証する。

**TC-301-E03**: 新パス 1 に "New456!"、新パス 2 に "Different!" を入力。`Error: Passwords do not match` が出力され vault 無変更を検証する。

**TC-301-E04**: 出力ディスクを意図的に満杯に近づける (またはモックで `fs::write` を失敗させる)。`vault.pqd.tmp` が削除され、旧 `vault.pqd` のバイト列が変更前と一致することを検証する。TASK-0091 の挙動を CLI 層から確認する。

**TC-301-B01**: 旧パス "Same123!"、新パス 1 & 2 に同じ "Same123!" を入力。`Warning: New password is identical to old password.` が stderr に出力され、change-password が正常完了 (exit 0) することを検証する。再 unlock も同パスワードで成功する。

**TC-NFR-103-02**: `cli.claude = true` で `cmd_change_password` を呼び出し、`Error: change-password is not permitted with --claude` で `bail!` されることを検証する。vault に一切の変更がないことも確認する。

## 注意事項

- TASK-0091 完了後に着手すること (再暗号化コアロジックに依存)
- `--password` フラグや `PQ_DIARY_PASSWORD` 環境変数は **旧パスワードにのみ** 適用し、新パスワードは常に TTY × 2 を強制する設計が安全 (詳細は TDD-red で確定)
- 既存 `not_implemented("change-password", "Sprint 3")` メッセージは完全削除する (REQ-703 の "Planned for Phase 2" 統一とは別、本コマンドは実装するため削除)
- 新旧同一は REQ-314 の意図通り「警告のみで処理続行」とする (エラー停止ではない)
- SIGINT テスト (TC-301-E05) はプラットフォーム依存が大きいため `#[ignore]` 付きで実装し、手動検証で確認することも許容
- パフォーマンステスト (TC-301-05) は TASK-0091 と同様 fast_params() を使用する
- メモリ安全性: `SecretString` の `expose_secret()` は比較のためにのみ呼び出し、戻り値を保存しない (Drop で自動 zeroize される)
- `Cli` 構造体の `password` フィールドは S9 で `Option<SecretString>` に変更済 (TC-S9-SEC-04) なので、本タスクではそれを流用する

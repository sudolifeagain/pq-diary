## タスク概要

未実装スケルトンである `legacy` / `legacy-access` / `daemon` の clap サブコマンド定義に `#[command(hide = true)]` を付与し、`pq-diary --help` のトップレベルヘルプから除外する。あわせて `dispatch` 内に残存する `not_implemented()` 呼び出しのメッセージを "Sprint 9" / "Sprint 10" 等の不揃いな表記から "Phase 2" に統一する。

設計ヒアリング (2026-05-17) で「Phase 2 以降に再設計予定の未実装スケルトンは clap 定義から削除せず hide 化のみで対応する。明示的に `pq-diary legacy --help` と呼び出した場合は依然として動作する」と確定済み。

`Commands::Init` / `Sync` / `ChangePassword` / `Info` / `Export` の "Planned for Sprint X" メッセージは TASK-0089 / 0090 / 0092 / 0093 / 0094 で本実装に置き換えられて消去されるが、本タスクでも残存チェックを行う。

## 実装詳細

### 1. clap Subcommand に `#[command(hide = true)]` を付与

**信頼性**: 🔵 *REQ-702, 設計ヒアリング 2026-05-17*

`cli/src/main.rs` の `Commands` enum において、以下 3 バリアントの直上に `#[command(hide = true)]` 属性を追加する:

```rust
// 該当箇所 (現状の行番号は参考値)
#[command(hide = true)]
Legacy {
    #[command(subcommand)]
    subcommand: LegacyCommands,
},

#[command(hide = true)]
LegacyAccess,

#[command(hide = true)]
Daemon {
    #[command(subcommand)]
    subcommand: DaemonCommands,
},
```

- `hide = true` は clap (derive) の標準属性で、`pq-diary --help` のサブコマンド一覧から該当エントリを抑制する
- ただしサブコマンド自体は登録されたままなので、`pq-diary legacy --help` や `pq-diary daemon start` のような明示的呼び出しは依然として動作する (TC-702-02 の前提)
- 子サブコマンド (`LegacyCommands`, `DaemonCommands`) には hide 属性を付与しない (親が hide なので親経由ではヘルプに出ない)

### 2. `not_implemented()` 呼び出しメッセージの "Phase 2" 統一

**信頼性**: 🔵 *REQ-703, 設計ヒアリング 2026-05-17*

`dispatch` 関数内の `not_implemented(cmd_name, sprint)` 呼び出しを、第 2 引数が "Phase 2" になるように書き換える。現状の対象 8 箇所:

| 箇所 | 現状 | 変更後 |
|------|------|--------|
| `Commands::Init` | `not_implemented("init", "Sprint 2")` | TASK-0089 で本実装に置換予定。本タスクでは残存チェック |
| `Commands::Sync` | `not_implemented("sync", "Sprint 8")` | TASK-0090 で本実装に置換予定。本タスクでは残存チェック |
| `Commands::Export` | `not_implemented("export", "Sprint 5")` | TASK-0094 で本実装に置換予定。本タスクでは残存チェック |
| `Commands::ChangePassword` | `not_implemented("change-password", "Sprint 3")` | TASK-0092 で本実装に置換予定。本タスクでは残存チェック |
| `Commands::Info { .. }` | `not_implemented("info", "Sprint 2")` | TASK-0093 で本実装に置換予定。本タスクでは残存チェック |
| `LegacyCommands::Init` | `not_implemented("legacy init", "Sprint 9")` | `not_implemented("legacy init", "Phase 2")` |
| `LegacyCommands::Rotate` | `not_implemented("legacy rotate", "Sprint 9")` | `not_implemented("legacy rotate", "Phase 2")` |
| `LegacyCommands::Set` | `not_implemented("legacy set", "Sprint 9")` | `not_implemented("legacy set", "Phase 2")` |
| `LegacyCommands::List` | `not_implemented("legacy list", "Sprint 9")` | `not_implemented("legacy list", "Phase 2")` |
| `Commands::LegacyAccess` | `not_implemented("legacy-access", "Sprint 9")` | `not_implemented("legacy-access", "Phase 2")` |
| `DaemonCommands::Start` | `not_implemented("daemon start", "Sprint 10")` | `not_implemented("daemon start", "Phase 2")` |
| `DaemonCommands::Stop` | `not_implemented("daemon stop", "Sprint 10")` | `not_implemented("daemon stop", "Phase 2")` |
| `DaemonCommands::Status` | `not_implemented("daemon status", "Sprint 10")` | `not_implemented("daemon status", "Phase 2")` |
| `DaemonCommands::Lock` | `not_implemented("daemon lock", "Sprint 10")` | `not_implemented("daemon lock", "Phase 2")` |

`not_implemented()` 関数は `Planned for {sprint}` を含むエラーメッセージを生成する設計のため (S9 TASK-0080 で確定)、引数を "Phase 2" に揃えることで最終メッセージが "Planned for Phase 2" に統一される。

**TASK-0089 / 0090 / 0092 / 0093 / 0094 との関係**:
本タスクの実施タイミングが上記 5 タスクより先になる場合、`Commands::Init` 等の `not_implemented("init", "Sprint 2")` も同様に "Phase 2" へ書き換えること。後発の場合は当該行はすでに本実装に置き換わっているため、残存している `not_implemented()` 呼び出しのみを対象とする (Phase 2 に統一されているか目視確認)。

## 注意事項

- `hide = true` は clap 4.x の derive 属性。古い clap (3.x) では `setting(AppSettings::Hidden)` が必要だが、本プロジェクトは clap 4.x 系を使用しているため `#[command(hide = true)]` で OK
- `legacy` / `daemon` の clap 定義自体は削除しないこと。Phase 2 で再設計するため定義は残し、ヘルプ表示のみを抑制する方針 (設計ヒアリング 2026-05-17 確定)
- `Commands::Init` 等の "Sprint X" → "Phase 2" 書き換えは、本タスクと TASK-0089 / 0090 / 0092 / 0093 / 0094 のどちらが先になっても整合性が取れるよう、実施時点での残存呼び出しを対象とすること
- テストコード内の `not_implemented("test-cmd", "S99")` (S9 TASK-0080 由来) は対象外。あくまで `dispatch` 内の本番コードのみを変更する
- Rust 規約遵守: 本タスクは clap derive 属性追加と文字列リテラル変更のみで、`unsafe` / `unwrap` / `zeroize` 規約には抵触しない

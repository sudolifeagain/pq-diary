## タスク概要

S11 で初めて macOS / Linux aarch64 で CI を走らせる。S10 hotfix で Linux x86_64 固有バグ (nix mlock API) が遅れて発覚した実例があるため、これらの新環境で同様のバグが出る可能性が高い。

発覚したバグは S11 内で修正するか、規模が大きい場合は S12 へ繰り越す。

## 完了条件 (バグが発覚した場合)

- [ ] 各バグについて修正コミットが含まれる
- [ ] 修正後の CI で全 8 ジョブ (check 3 + audit 1 + smoke 4) が green
- [ ] 修正規模が大きすぎる場合は PR 説明に「S12 繰り越し理由」を明記

## 完了条件 (バグが発覚しなかった場合)

- [ ] PR 説明に「全 OS 一発 pass、本タスクは該当なし」と明記

## 想定される発覚バグの種類

### macOS 固有
- `dirs::home_dir()` の挙動差 (`$HOME` 解決)
- 一時ディレクトリパス (`/tmp` vs `/var/folders/...`)
- `nix::sys::resource::getrlimit` の RLIMIT_CORE 既定値差
- `/proc/self/status` 不在 (Linux 専用) で `query_mlock_active` / `query_coredump_disabled` の挙動

### Linux aarch64 固有
- x86_64 intrinsic を使ったコードがあれば fail (現状なし、要確認)
- ml-kem / ml-dsa の aarch64 最適化パスでテストデータ差
- nix クレートの aarch64 ABI 差

### Windows 固有
- 既存テストで pass 済みのため変更影響なし想定

## 実装詳細 (バグ発覚時のフロー)

1. CI ログから失敗箇所と OS を特定
2. ローカルでクロスコンパイル可能なら再現 (`cargo build --target aarch64-unknown-linux-gnu` 等)
3. 再現不可なら CI 上で `tmate` action 等で debug (最終手段)
4. 修正 → push → CI 再確認
5. 修正規模次第:
   - 1 ファイル 数十行: S11 で対応
   - モジュール rewrite 級: S12 で別タスク化、本 TASK-0103 は「方針提示のみ」で完了

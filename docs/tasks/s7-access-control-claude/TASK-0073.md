## 実装詳細

### 1. 統合テスト: vault CRUD サイクル

**信頼性**: 🔵 *REQ-030~037*

vault create → vault list → vault policy → vault delete の完全なCRUDサイクルを検証する:

1. vault create で新しい vault を作成
2. vault list で作成された vault が一覧に表示されることを確認
3. vault policy でポリシーを変更し、変更が反映されることを確認
4. vault delete で vault を削除し、一覧から消えることを確認

### 2. 統合テスト: --claude + None vault の即座拒否

**信頼性**: 🔵 *REQ-101, NFR-101*

--claude フラグ付きで None ポリシーの vault に対して操作を実行し、復号処理が行われずに即座に拒否されることを検証する。パスワードプロンプトが表示されないことを確認する。

### 3. 統合テスト: --claude + WriteOnly vault の操作制御

**信頼性**: 🔵 *REQ-020~025*

--claude フラグ付きで WriteOnly ポリシーの vault に対して:
- 書き込み操作（new, edit, delete 等）が許可されることを確認
- 読み取り操作（list, show, search, stats 等）が拒否されることを確認

### 4. 統合テスト: --claude + Full vault の全操作許可

**信頼性**: 🔵 *REQ-020~025*

--claude フラグ付きで Full ポリシーの vault に対して、全操作（読み取り・書き込み）が許可されることを検証する。

### 5. 統合テスト: vault delete --zeroize

**信頼性**: 🔵 *REQ-037*

vault delete --zeroize を実行し、ファイルが上書き削除されていることを検証する。削除後にファイルが存在しないことを確認する。

### 6. 統合テスト: 後方互換性

**信頼性**: 🔵 *EDGE-001*

既存の vault.toml に `policy="none"` 文字列が含まれている場合、AccessPolicy::None として正しく読み込まれることを検証する。古いフォーマットの vault.toml との後方互換性を確認する。

### 7. doc comments 追加

**信頼性**: 🔵 *Rust ドキュメント規約*

Sprint 7 で追加した全新規公開型・関数に `///` doc comments を追加する。英語で記述する。

**対象ファイル**:

- **core/src/policy.rs**: `AccessPolicy`, `OperationType`, `PolicyDecision`, `check_access()`, `classify_operation()`
- **core/src/vault/init.rs**: VaultManager の新メソッド（create_vault, list_vaults_with_policy, set_policy, delete_vault）
- **core/src/lib.rs**: DiaryCore の新アクセサ

**doc comment 規約**:
- 1行目: 型/関数の概要（英語、1文）
- 2行目以降: 詳細説明、引数 (`# Arguments`)、戻り値 (`# Returns`)、エラー (`# Errors`)
- 主要な公開APIには `# Examples` セクション

### 8. NFR パフォーマンスチェック: None ポリシー拒否 < 10ms

**信頼性**: 🔵 *NFR-101*

None ポリシーでの拒否が 10ms 以内に完了することを検証する（TC-S7-NFR-001-01）。vault.toml の読み込みとポリシー評価のみで、復号処理を行わないため高速であることを確認する。

### 9. NFR パフォーマンスチェック: vault create < 3s

**信頼性**: 🔵 *NFR-201*

vault create コマンドが 3秒以内に完了することを検証する（TC-S7-NFR-003-01）。既存の init パフォーマンステストでカバー済の場合は追加テスト不要。

### 10. vault.toml 改ざん検出

**信頼性**: 🔵 *NFR-103*

vault.toml を手動で不正な値に改ざんした場合、DiaryError::Config が返ることを検証する（TC-S7-NFR-103-01）。

### 11. 最終品質チェック

**信頼性**: 🔵 *開発フロー標準*

```bash
cargo test --workspace
cargo clippy --workspace -- -D warnings
cargo fmt --check
cargo doc --workspace --no-deps
```

全コマンドがエラー・警告なしで完了することを確認する。

### テスト詳細

**TC-S7-INT-01**: vault を新規作成し、list で表示を確認、policy で変更を確認、delete で削除を確認する完全なCRUDサイクルテスト。各操作後の状態遷移が正しいことを検証する。

**TC-S7-INT-02**: --claude フラグ付きで None ポリシーの vault に対して new コマンドを実行し、パスワードプロンプトが表示されずに DiaryError::Policy で即座に拒否されることを検証する。

**TC-S7-INT-03**: --claude フラグ付きで WriteOnly ポリシーの vault に対して、new（Write）が許可され、list（Read）が拒否されることを検証する。

**TC-S7-INT-04**: --claude フラグ付きで Full ポリシーの vault に対して、new（Write）と list（Read）の両方が許可されることを検証する。

**TC-S7-INT-05**: vault delete --zeroize を実行し、vault ディレクトリとファイルが完全に削除されていることを検証する。

**TC-S7-INT-06**: `[access]` セクションに `policy = "none"` を持つ既存の vault.toml を手動作成し、VaultConfig::from_file() で AccessPolicy::None として正しくパースされることを検証する。

**TC-S7-NFR-001-01**: None ポリシーでの拒否処理の実行時間を計測し、10ms 以内であることをアサートする。`std::time::Instant` で計測する。

**TC-S7-NFR-003-01**: vault create コマンドの実行時間を計測し、3秒以内であることをアサートする。既存の init パフォーマンステストでカバー済であれば確認のみ。

**TC-S7-NFR-103-01**: vault.toml の policy フィールドを不正な値（例: `policy = "hacked"`）に手動で書き換え、VaultConfig::from_file() が DiaryError::Config を返すことを検証する。

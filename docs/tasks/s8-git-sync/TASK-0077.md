## タスク概要

core/src/git.rs に `git_pull_merge()` 関数を実装する。リモートから fetch し、ローカルとリモートの vault.pqd をエントリ単位で UUID + content_hmac により 3-way マージする。コンフリクト時は last-write-wins（updated_at 比較）で解決し、claude_mode ではローカル優先とする。

## 実装詳細

### 1. git_pull_merge() 関数シグネチャ (core/src/git.rs)

**信頼性**: 🔵 *REQ-020*

`git_pull_merge()` 関数を以下のシグネチャで定義する:

```rust
pub fn git_pull_merge(
    vault_dir: &Path,
    config: &GitSyncConfig,
    engine: &CryptoEngine,
    vault_path: &Path,
    claude_mode: bool,
) -> Result<MergeResult, DiaryError>
```

### 2. MergeResult / MergeConflict / ConflictResolution 型定義 (core/src/git.rs)

**信頼性**: 🔵 *REQ-020, REQ-026*

マージ結果を表す構造体と関連型を定義する:

```rust
#[derive(Debug, Clone)]
pub struct MergeResult {
    pub added: usize,
    pub updated: usize,
    pub deleted: usize,
    pub conflicts: Vec<MergeConflict>,
}

#[derive(Debug, Clone)]
pub struct MergeConflict {
    pub uuid: [u8; 16],
    pub local: EntryRecord,
    pub remote: EntryRecord,
}

#[derive(Debug, Clone, Copy, PartialEq, Eq)]
pub enum ConflictResolution {
    KeepLocal,
    KeepRemote,
}
```

### 3. バックアップ作成と復元 (core/src/git.rs)

**信頼性**: 🔵 *REQ-028*

マージ処理の安全性を確保するため、ローカル vault.pqd のバックアップを作成する:

1. ローカル vault.pqd を vault.pqd.bak にコピー
2. マージ処理中にエラーが発生した場合、バックアップから復元する
3. マージ成功後にバックアップファイルを削除する

### 4. git fetch + リモート変更の検出 (core/src/git.rs)

**信頼性**: 🔵 *REQ-020, REQ-021*

リモートからの変更を取得し、変更の有無を判定する:

1. `git fetch origin` を実行
2. `git rev-list HEAD..origin/main --count` でリモートの新規コミット数を取得
3. カウントが 0 の場合 → バックアップを復元し、`MergeResult` を全フィールド 0 で返す（no-op）
4. カウントが 1 以上の場合 → マージ処理を続行

### 5. リモート vault.pqd の取得 (core/src/git.rs)

**信頼性**: 🔵 *REQ-022*

リモートの vault.pqd を一時的にチェックアウトして読み取る:

1. `git checkout origin/main -- vault.pqd vault.toml` でリモート版を取得
2. `read_vault()` でリモート版の vault.pqd を読み取り → `remote_entries: Vec<EntryRecord>`
3. バックアップからローカル vault.pqd を復元
4. `read_vault()` でローカル版の vault.pqd を読み取り → `local_entries: Vec<EntryRecord>`

### 6. 3-way マージアルゴリズム (core/src/git.rs)

**信頼性**: 🔵 *REQ-023, REQ-024, REQ-025*

UUID + content_hmac に基づくエントリ単位のマージアルゴリズム:

1. ローカルとリモートの両方について `HashMap<[u8; 16], EntryRecord>` を構築する（キーは UUID）
2. **ローカルのみ**: ローカルの HashMap に存在しリモートにない UUID → そのまま保持（ローカルで新規作成されたエントリ）
3. **リモートのみ**: リモートの HashMap に存在しローカルにない UUID → マージ結果に追加（リモートで新規作成されたエントリ）→ `added` カウント増加
4. **両方に存在**: 同一 UUID が両方に存在する場合:
   a. `content_hmac` が同一 → 変更なし、ローカルを保持
   b. `content_hmac` が異なる → `updated_at` タイムスタンプ（u64 UNIX タイムスタンプ）を比較し、値が大きい方（新しい方）を採用（last-write-wins）→ `updated` カウント増加
   c. `updated_at` が同一 → 真のコンフリクト、解決が必要

### 7. コンフリクト解決 (core/src/git.rs)

**信頼性**: 🔵 *REQ-026*

`updated_at` が同一で真のコンフリクトが発生した場合の解決戦略:

- `claude_mode = true` の場合 → 自動的にローカルを採用（`ConflictResolution::KeepLocal`）
- `claude_mode = false` の場合 → `MergeConflict` を `MergeResult.conflicts` ベクタに格納して返す。CLI 層でユーザーに解決方法を選択させる

### 8. マージ結果の書き込み (core/src/git.rs)

**信頼性**: 🔵 *REQ-028*

マージされたエントリ群を vault.pqd にアトミックに書き込む:

1. マージ済みの `Vec<EntryRecord>` を構築
2. `write_vault()` でアトミック書き込み（一時ファイル経由、`core/src/vault/writer.rs` の既存実装を使用）
3. プライバシー設定を適用して git add + commit を実行（マージコミット）

### 9. バックアップファイルのクリーンアップ (core/src/git.rs)

**信頼性**: 🔵 *REQ-028*

マージが正常に完了した後、vault.pqd.bak を削除する。エラー発生時はバックアップを残して復旧に使用できるようにする。

### 10. content_hmac の比較に関する注意事項 (core/src/git.rs)

**信頼性**: 🔵 *REQ-023*

`content_hmac` は暗号文に対して計算される（平文ではない）。同一の平文であっても異なる IV で暗号化された場合は異なる暗号文となり、異なる HMAC が生成される。したがって HMAC の比較は「生のレコードが変更されたか」を検出するものであり、「意味的な内容が変更されたか」を検出するものではない。last-write-wins マージではこの動作で問題ない。

## 注意事項

- `EntryRecord.uuid` は `[u8; 16]` 型。比較時は `uuid::Uuid::from_bytes()` を使用すること
- `content_hmac` は `[u8; 32]` 型。バイト列の直接比較 (`==`) で判定する
- `updated_at` は `u64` 型（UNIX タイムスタンプ）。値が大きい方が新しい
- `content_hmac` は暗号文に対して計算される（平文ではない）。同一平文でも異なる IV で暗号化されると異なる HMAC になる。これは last-write-wins マージにおいて期待通りの動作である
- マージ後の再暗号化には `CryptoEngine` が必要。ただしマージは暗号化された `EntryRecord` 単位で操作するため、個々のエントリの復号は不要。レコードをそのまま差し替える
- テストでは `tempfile::tempdir()` を使用してローカルとリモートの git リポジトリを一時ディレクトリに作成する

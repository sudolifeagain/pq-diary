## 実装詳細

### 1. 統合テスト: E2E フロー (git-init → git-push → git-pull → マージ検証)

**信頼性**: 🔵 *REQ-001~003, REQ-030*

完全なエンドツーエンドフローを検証する:

1. テスト用 vault を作成する
2. git-init で .git を初期化する
3. エントリを追加し git-push を実行する
4. 第2の git リポジトリを "リモート" としてシミュレートし、リモート側で変更を行う
5. git-pull を実行しマージ結果を検証する
6. ローカルのエントリが正しく統合されていることを確認する

### 2. 統合テスト: git-init が .git + .gitignore + vault.toml author フィールドを作成する

**信頼性**: 🔵 *REQ-001, REQ-016~017*

git-init 実行後の状態を検証する:

1. .git ディレクトリが作成されていること
2. .gitignore にプライバシー保護用のルール（entries/*.md 等）が含まれていること
3. vault.toml の author フィールドが匿名値で設定されていること

### 3. 統合テスト: git-push プライバシー保護

**信頼性**: 🔵 *REQ-022, REQ-025~026*

git-push がプライバシーを保護していることを検証する:

1. コミットの author が匿名であること（実名が含まれないこと）
2. コミットメッセージが固定文字列であること（エントリ内容が漏洩しないこと）
3. タイムスタンプファジングが適用されていること（正確な時刻が漏洩しないこと）

### 4. 統合テスト: git-pull マージ — ローカルのみ保持、リモートのみ追加、競合は updated_at で解決

**信頼性**: 🔵 *REQ-030~031*

マージロジックの各ケースを検証する:

1. ローカルにのみ存在するエントリはそのまま保持されること
2. リモートにのみ存在するエントリが追加されること
3. 両方に存在し内容が異なるエントリは updated_at の新しい方で解決されること

### 5. 統合テスト: git-sync = pull → push

**信頼性**: 🔵 *REQ-005, REQ-030*

git-sync コマンドが pull → push の順序で実行されることを検証する:

1. リモートに変更がある状態で git-sync を実行する
2. リモートの変更がローカルに統合されていることを確認する
3. ローカルの変更がリモートにプッシュされていることを確認する

### 6. 統合テスト: git-status が git status をラップする

**信頼性**: 🔵 *REQ-004*

git-status コマンドが git status の出力を正しくラップしていることを検証する:

1. 変更なしの状態で "clean" 相当の出力が返ること
2. 変更ありの状態で変更内容が出力に含まれること

### 7. 統合テスト: --claude 競合解決 — ローカル自動優先

**信頼性**: 🔵 *REQ-040*

--claude フラグ付きで競合が発生した場合、インタラクティブプロンプトなしでローカル側が自動的に優先されることを検証する。

### 8. 統合テスト: エラーケース

**信頼性**: 🔵 *EDGE ケース全般*

各エラーケースが適切に処理されることを検証する:

1. git コマンド未インストール時に適切なエラーメッセージが返ること
2. .git 未初期化時に push/pull/sync が適切なエラーで失敗すること
3. リモート未設定時に push/pull が適切なエラーで失敗すること
4. 既に初期化済の vault に対する git-init が適切にハンドリングされること

### 9. doc comments 追加

### テスト詳細

**TC-S8-INT-01**: テスト用 vault を作成し、git-init → エントリ追加 → git-push → 第2リポジトリでリモート変更をシミュレート → git-pull → マージ結果検証の完全な E2E フローテスト。各操作後の状態遷移が正しいことを検証する。

**TC-S8-INT-02**: git-init 実行後に .git ディレクトリ、.gitignore ファイル、vault.toml の author フィールドが正しく設定されていることを検証する。.gitignore に entries/*.md が含まれていることを確認する。

**TC-S8-INT-03**: git-push 実行後のコミット履歴を検査し、author が匿名であること、メッセージが固定文字列であること、タイムスタンプが正確な時刻でないことを検証する。

**TC-S8-INT-04**: ローカルとリモートで異なるエントリ群を作成し、git-pull 実行後にローカルのみのエントリが保持され、リモートのみのエントリが追加され、両方に存在するエントリは updated_at の新しい方で解決されることを検証する。

**TC-S8-INT-05**: リモートに変更がある状態で git-sync を実行し、リモート変更がローカルに反映され、ローカル変更がリモートにプッシュされていることを検証する。

**TC-S8-INT-06**: 変更なし・変更ありの2状態で git-status を実行し、適切な出力が返されることを検証する。

**TC-S8-INT-07**: --claude フラグ付きで両方に同一エントリの異なるバージョンが存在する状態で git-pull を実行し、インタラクティブプロンプトなしでローカル側が優先されることを検証する。

**TC-S8-INT-E01**: git コマンドが存在しない環境をシミュレートし、全 git 系コマンドが "git not found" 相当のエラーで失敗することを検証する。

**TC-S8-INT-E02**: .git ディレクトリが存在しない vault に対して push/pull/sync を実行し、"git not initialized" 相当のエラーが返ることを検証する。

**TC-S8-INT-E03**: リモートが設定されていない状態で push/pull を実行し、"no remote configured" 相当のエラーが返ることを検証する。

**TC-S8-INT-E04**: 既に git-init 済の vault に対して再度 git-init を実行した場合の動作を検証する。エラーにならず冪等に処理されることを確認する。

**TC-S8-NFR-001-01**: check_git_available() の実行時間を `std::time::Instant` で計測し、100ms 以内であることをアサートする。

**TC-S8-SEC-01**: git-push 後の staging area に entries/*.md ファイルが含まれていないことを git ls-files で検証する。

**TC-S8-SEC-02**: git-push 後のコミットログを検査し、author フィールドとコミットメッセージにパスワード、エントリ内容、個人名が含まれていないことを検証する。

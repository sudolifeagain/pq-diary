## タスク概要

core/src/crypto/secure_mem.rs にメモリロック機能を追加する。鍵素材バッファを OS のメモリロック機構（Unix: mlock、Windows: VirtualLock）でページアウト防止し、スワップ領域への秘密鍵の書き出しを防ぐ。mlock 失敗時は警告を出力して処理を継続するフェイルソフト設計とする。

## 実装詳細

### 1. mlock_buffer / munlock_buffer 関数 (core/src/crypto/secure_mem.rs)

**信頼性**: 🔵 *REQ-001, REQ-002, REQ-003, REQ-005*

プラットフォームごとにメモリロック/アンロック関数を実装する:

#### Unix 版 (mlock)

```rust
#[cfg(unix)]
pub fn mlock_buffer(buf: &[u8]) -> Result<(), DiaryError> {
    unsafe { nix::sys::mman::mlock(buf.as_ptr() as *const _, buf.len()) }
        .map_err(|e| DiaryError::Crypto(format!("mlock failed: {e}")))
}

#[cfg(unix)]
pub fn munlock_buffer(buf: &[u8]) -> Result<(), DiaryError> {
    unsafe { nix::sys::mman::munlock(buf.as_ptr() as *const _, buf.len()) }
        .map_err(|e| DiaryError::Crypto(format!("munlock failed: {e}")))
}
```

#### Windows 版 (VirtualLock)

```rust
#[cfg(windows)]
pub fn mlock_buffer(buf: &[u8]) -> Result<(), DiaryError> {
    let result = unsafe {
        windows_sys::Win32::System::Memory::VirtualLock(
            buf.as_ptr() as *mut _,
            buf.len(),
        )
    };
    if result == 0 {
        Err(DiaryError::Crypto("VirtualLock failed".into()))
    } else {
        Ok(())
    }
}

#[cfg(windows)]
pub fn munlock_buffer(buf: &[u8]) -> Result<(), DiaryError> {
    let result = unsafe {
        windows_sys::Win32::System::Memory::VirtualUnlock(
            buf.as_ptr() as *mut _,
            buf.len(),
        )
    };
    if result == 0 {
        Err(DiaryError::Crypto("VirtualUnlock failed".into()))
    } else {
        Ok(())
    }
}
```

#### 非対応プラットフォーム (no-op)

```rust
#[cfg(not(any(unix, windows)))]
pub fn mlock_buffer(_buf: &[u8]) -> Result<(), DiaryError> { Ok(()) }

#[cfg(not(any(unix, windows)))]
pub fn munlock_buffer(_buf: &[u8]) -> Result<(), DiaryError> { Ok(()) }
```

### 2. CryptoEngine unlock/lock との統合

**信頼性**: 🔵 *REQ-001, REQ-002, REQ-004*

CryptoEngine のアンロック/ロック処理にメモリロック操作を組み込む:

- **unlock_with_vault 後**: MasterKey 構築後に sym_key、kem_sk、dsa_sk の各バッファに対して mlock_buffer() を呼び出す
- **mlock 失敗時**: `eprintln!("warning: failed to lock key material in memory: {e}")` で警告を出力し、処理を継続する (REQ-004)
- **lock() 実行時**: バッファを drop する前に munlock_buffer() を呼び出してメモリロックを解放する (REQ-003)

```rust
// unlock 後
if let Err(e) = mlock_buffer(sym_key.as_ref()) {
    eprintln!("warning: failed to lock key material in memory: {e}");
}

// lock 時
let _ = munlock_buffer(sym_key.as_ref());
// その後 drop
```

### 4. unsafe 使用の正当性

**信頼性**: 🔵 *CLAUDE.md 規約*

本タスクでの unsafe 使用は以下のシステムコールに限定する:

- `nix::sys::mman::mlock` — メモリロック (Unix)
- `nix::sys::mman::munlock` — メモリアンロック (Unix)
- `windows_sys::Win32::System::Memory::VirtualLock` — メモリロック (Windows)
- `windows_sys::Win32::System::Memory::VirtualUnlock` — メモリアンロック (Windows)

CLAUDE.md の unsafe 許可範囲 (mlock/VirtualLock) に合致する。

### テスト詳細

**TC-S9-083-01**: テスト用の小さなバッファ（例: 32バイト）を確保し、`mlock_buffer()` を呼び出して `Ok(())` が返ることを検証する。テスト実行プラットフォーム (Windows) で VirtualLock が成功することを確認する。

**TC-S9-083-02**: バッファに対して `mlock_buffer()` を呼び出した後、`munlock_buffer()` を呼び出して `Ok(())` が返ることを検証する。ロックとアンロックの対称性を確認する。

**TC-S9-083-03**: mlock 失敗シナリオを検証する。RLIMIT_MEMLOCK 制限により mlock が失敗しうる環境では、エラーが `DiaryError::Crypto` として返されることを確認する。CryptoEngine 統合時には eprintln 警告が出力されるが処理が継続することを検証する。

**TC-S9-083-04**: CryptoEngine の unlock → lock サイクルを実行し、鍵素材バッファに対してメモリロックが適用されていることを検証する。lock 後にバッファが正常に解放されることを確認する。

**TC-S9-083-05**: `#[cfg(not(any(unix, windows)))]` の no-op 実装をコンパイル検証する。実行環境が Windows のため、直接実行テストは Unix/Windows 版で実施する。

**TC-S9-083-06**: mlock されていないバッファに対して `munlock_buffer()` を呼び出した場合の動作を検証する。エラーが返る可能性はあるが、パニックしないことを確認する。

**TC-S9-083-07**: 長さ0のバッファに対して `mlock_buffer()` を呼び出した場合の動作を検証する。パニックせず、`Ok(())` またはエラーが返ることを確認する。

## 注意事項

- nix クレートには "mman" フィーチャーを追加すること。不要なフィーチャーは含めない
- Windows では windows-sys に `Win32_System_Memory` フィーチャーが必要
- mlock は RLIMIT_MEMLOCK 制限のあるシステムで失敗しうる。テスト環境でも失敗する可能性があるため、`#[ignore]` 属性の付与を検討すること
- テストは Windows のみ (現在の開発環境) で実行検証。Unix コードは cfg による コンパイル検証
- unsafe 使用は CLAUDE.md の許可範囲 (mlock/VirtualLock) に厳密に限定すること
- 既存の `DiaryError::Crypto(String)` バリアントを使用すること。新しいエラーバリアントを追加しないこと

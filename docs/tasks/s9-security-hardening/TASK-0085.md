## 実装詳細

### 1. E2E テスト: vault ライフサイクル (cli/src/commands.rs tests)

**信頼性**: 🔵 *REQ-030~035*

完全な vault ライフサイクルのエンドツーエンドフローを検証する:

1. **init → new → list → show → edit → delete → search → stats**: vault 初期化からエントリ作成、一覧表示、詳細表示、編集、削除、検索、統計情報取得までの全フローを単一テストで実行する
2. **テンプレートライフサイクル**: template add → list → show → delete の全フローを検証する
3. **インポートライフサイクル**: テスト用 Markdown ファイルを作成 → import コマンドで取り込み → エントリとして正しく登録されていることを検証する
4. **vault 管理**: vault create → list → policy → delete の全フローを検証する
5. **Git ライフサイクル**: git-init → git-push → git-status の各コマンドが正常に動作することを検証する（テスト環境で可能な範囲）
6. **ポリシー適用**: --claude フラグと none/write_only/full の各ポリシーでコマンドが適切に制御されることを検証する

### 2. パフォーマンステスト

**信頼性**: 🔵 *NFR-001~005*

各操作のパフォーマンスターゲットを検証する。テスト安定性のために Argon2 の fast_params() を使用する:

| 操作 | ターゲット | 備考 |
|------|-----------|------|
| init (vault 作成) | < 5s | fast_params() でテスト安定性確保 |
| unlock | < 5s | fast_params() でテスト安定性確保 |
| new entry | < 200ms | エントリ作成単体 |
| list 100 entries | < 500ms | 事前に100エントリを作成 |
| lock | < 50ms | vault ロック単体 |

- `std::time::Instant` で各操作の実行時間を計測する
- Argon2 パラメータにはテスト用の fast_params() を使用し、S8 で発生したタイミング不安定問題を回避する
- 本番用パラメータでのパフォーマンスは手動検証とする

### 3. セキュリティ統合テスト

**信頼性**: 🔵 *技術的負債検証*

S8 までに修正された技術的負債が正しく機能していることを検証する:

1. **verify_hmac が Result<bool, DiaryError> を返す** (M-5 負債検証): verify_hmac の戻り値型が正しいことを確認する
2. **エントリ読み取り時の HMAC 検証**: 改竄された暗号文でエントリを読み取ろうとした場合にエラーが返ることを検証する
3. **not_implemented が exit ではなくエラーを返す**: not_implemented() 関数が `std::process::exit` ではなく `Result::Err` を返すことを検証する
4. **Cli.password が SecretString 型**: CLI のパスワードフィールドが `secrecy::SecretString` 型であることを型システムで保証する

### テスト詳細

**TC-S9-E2E-01**: テスト用 vault を作成し、エントリの新規作成 → 一覧表示でエントリが存在すること → 詳細表示で内容が正しいこと → 編集で内容が更新されること → 削除でエントリが消えること → 検索で期待結果が返ること → 統計情報が正確であることを順次検証する。

**TC-S9-E2E-02**: テスト用 vault にテンプレートを追加し、テンプレート一覧に表示されること → テンプレート詳細が正しいこと → テンプレート削除後に一覧から消えることを検証する。

**TC-S9-E2E-03**: テスト用ディレクトリに Obsidian 互換の Markdown ファイル（`[[wiki-link]]` や `#ネスト/タグ` を含む）を作成し、import コマンドで取り込み、pq-diary 形式に正しく変換されたエントリが存在することを検証する。

**TC-S9-E2E-04**: vault create で新しい vault を作成し、vault list で一覧に表示されること → vault policy で設定が変更されること → vault delete で削除されることを検証する。

**TC-S9-E2E-05**: vault 内で git-init を実行し .git が作成されること → git-push でコミットが作成されること → git-status で状態が表示されることを検証する。テスト環境の制約で実リモートリポジトリは使用しない。

**TC-S9-E2E-06**: --claude フラグを付与し、none ポリシーでは読み取り・書き込みが拒否されること、write_only ポリシーでは書き込みのみ許可されること、full ポリシーでは全操作が許可されることを検証する。

**TC-S9-PERF-01**: fast_params() を使用して vault を初期化し、初期化完了までの時間が 5 秒以内であることを `std::time::Instant` で計測・検証する。

**TC-S9-PERF-02**: fast_params() を使用して vault をアンロックし、アンロック完了までの時間が 5 秒以内であることを計測・検証する。

**TC-S9-PERF-03**: アンロック済 vault に対してエントリを新規作成し、作成完了までの時間が 200ms 以内であることを計測・検証する。

**TC-S9-PERF-04**: アンロック済 vault に 100 件のエントリを事前作成し、list コマンドの実行時間が 500ms 以内であることを計測・検証する。

**TC-S9-PERF-05**: アンロック済 vault に対して lock 操作を実行し、ロック完了までの時間が 50ms 以内であることを計測・検証する。

**TC-S9-SEC-01**: verify_hmac 関数を呼び出し、戻り値が `Result<bool, DiaryError>` 型であることをコンパイル時の型チェックで保証する。正常なエントリで Ok(true)、不一致で Ok(false) が返ることを検証する。

**TC-S9-SEC-02**: 正常に作成したエントリの暗号文を意図的に改竄し、エントリ読み取り時に HMAC 検証エラー (DiaryError) が返ることを検証する。

**TC-S9-SEC-03**: not_implemented() 関数を呼び出し、`std::process::exit` ではなく `Result::Err` が返ることを検証する。テストプロセスが終了しないことを確認する。

**TC-S9-SEC-04**: Cli 構造体の password フィールドが `Option<SecretString>` 型であることを型アサーションで検証する。生の `String` 型でないことを保証する。

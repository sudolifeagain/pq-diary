## タスク概要

CLI コマンドハンドラにおける vault のロック管理を VaultGuard drop guard パターンに移行する。現在の各コマンドでは手動で `core.lock()` を呼んでいるが、エラーパスでのロック解放漏れのリスクがある。RAII パターンの VaultGuard を導入し、スコープ離脱時に自動的に `lock()` が呼ばれることを保証する。

## 実装詳細

### 1. VaultGuard 構造体定義 (cli/src/commands.rs)

RAII ガードパターンを実装する:

```rust
struct VaultGuard<'a> {
    core: &'a DiaryCore,
}

impl<'a> VaultGuard<'a> {
    fn new(core: &'a DiaryCore) -> Result<Self> {
        core.unlock(/* ... */)?;
        Ok(Self { core })
    }
}

impl<'a> Drop for VaultGuard<'a> {
    fn drop(&mut self) {
        // lock() のエラーは無視 (drop 内では Result を返せない)
        let _ = self.core.lock();
    }
}

impl<'a> std::ops::Deref for VaultGuard<'a> {
    type Target = DiaryCore;
    fn deref(&self) -> &Self::Target {
        self.core
    }
}
```

### 2. 全コマンドハンドラの移行

以下の10コマンドを VaultGuard パターンに移行する:

1. `cmd_new`
2. `cmd_list`
3. `cmd_show_impl`
4. `cmd_edit_impl`
5. `cmd_delete_impl`
6. `cmd_template_add_impl`
7. `cmd_template_list`
8. `cmd_template_show`
9. `cmd_template_delete_impl`
10. `cmd_today_impl`

各コマンドの移行パターン:

```rust
// Before
fn cmd_new(core: &DiaryCore, /* ... */) -> Result<()> {
    core.unlock(/* ... */)?;
    // ... 処理 ...
    match result {
        Ok(_) => { core.lock()?; Ok(()) }
        Err(e) => { let _ = core.lock(); Err(e) }
    }
}

// After
fn cmd_new(core: &DiaryCore, /* ... */) -> Result<()> {
    let guard = VaultGuard::new(core)?;
    // ... guard 経由で処理 ...
    // drop 時に自動で lock() が呼ばれる
    Ok(())
}
```

### 3. 明示的 lock() 呼び出しの除去

移行完了後、全コマンドから明示的な `core.lock()` 呼び出しを除去する。`grep` で残存する `core.lock()` がないことを確認する。

### テスト詳細

**TC-A13-01**: VaultGuard を作成してスコープ外に出し、`lock()` が呼ばれたことを検証する。モックまたはフラグで `lock()` の呼び出しを検出する。

**TC-A13-02**: VaultGuard のスコープ内で意図的にエラーを発生させ (`?` 演算子で早期リターン)、それでも `lock()` が呼ばれることを検証する。Drop の RAII 保証により、パニックを含むあらゆる離脱パスで `lock()` が実行されることを確認する。

**TC-A13-03**: 既存の全テスト (cmd_new, cmd_list, cmd_show, cmd_edit, cmd_delete, cmd_template_*, cmd_today) が VaultGuard 移行後も通過することを検証する。リグレッションテストとして機能する。

## タスク概要

正規表現による全文検索のコアモジュールを実装する。ストリーミング方式（1エントリずつ復号→regex match→zeroize）でメモリ安全に検索を行い、body/title/tags/テンプレート本文の4フィールドを検索対象とする。マッチ行と前後N行のコンテキスト抽出機能を含む。

### 2. search モジュール新規作成 (core/src/search.rs)

**信頼性**: 🔵 *REQ-B01, REQ-B02, REQ-B03 + types.rs 参照*

以下の型を定義する（types.rs の設計に基づく）:

- **`SearchQuery`**: 検索クエリパラメータ
  - `pattern: String` — regex パターン文字列
  - `tag_filter: Option<String>` — タグ前方一致フィルタ
  - `context_lines: usize` — コンテキスト行数（デフォルト: 2）
  - `count_only: bool` — マッチ数のみ返却フラグ

- **`SearchMatch`**: 個別マッチ結果
  - `uuid_hex: String` — エントリUUID (hex)
  - `title: String` — エントリタイトル
  - `updated_at: u64` — 更新タイムスタンプ
  - `matched_field: String` — マッチフィールド ("body"/"title"/"tags"/"template")
  - `context_blocks: Vec<ContextBlock>` — コンテキストブロック

- **`ContextBlock`**: マッチ周辺行
  - `match_line_number: usize` — 1-based マッチ行番号
  - `lines: Vec<(usize, String, bool)>` — (行番号, 内容, is_match)

- **`SearchResults`**: 検索結果サマリー
  - `matches: Vec<SearchMatch>` — 全マッチ
  - `matched_entry_count: usize` — マッチエントリ数
  - `matched_line_count: usize` — マッチ行数

### 3. search_entries() 関数

**信頼性**: 🔵 *REQ-B08: 復号後インメモリ検索、REQ-B09: zeroize*

```rust
pub fn search_entries(
    vault_path: &Path,
    engine: &CryptoEngine,
    query: &SearchQuery,
) -> Result<SearchResults, DiaryError>
```

ストリーミング方式の処理フロー:
1. `regex::Regex::new(&query.pattern)?` でコンパイル（無効パターンは `DiaryError::Search` を返す）
2. `read_vault()` で vault ヘッダ + レコード一覧取得
3. 各レコードについて:
   a. `engine.decrypt(iv, ciphertext)` → `Zeroizing<Vec<u8>>`
   b. `serde_json::from_slice()` → `EntryPlaintext`
   c. `tag_filter` がある場合は `Tag::is_prefix_of()` でフィルタ
   d. body/title/tags/テンプレート本文に対して `regex.is_match()` / `regex.find_iter()`
   e. マッチあれば `format_context()` でコンテキスト抽出 → `SearchMatch` を結果に追加
   f. `EntryPlaintext` は `ZeroizeOnDrop` で自動 zeroize

### 4. format_context() 関数

**信頼性**: 🔵 *REQ-B04, REQ-B05: コンテキスト行表示*

```rust
fn format_context(
    body: &str,
    regex: &Regex,
    context_lines: usize,
) -> Vec<ContextBlock>
```

処理:
1. body を行分割
2. 各行に対して `regex.is_match()` でマッチ行を特定
3. マッチ行の前後 `context_lines` 行を含む `ContextBlock` を構築
4. 隣接するブロックはマージして重複行を排除

### 5. 検索対象フィールド

**信頼性**: 🔵 *REQ-B03: body, title, tags, テンプレート本文*

4フィールドそれぞれに対してマッチを検査:
- **body**: `regex.is_match(plaintext.body)` → context 付きマッチ
- **title**: `regex.is_match(plaintext.title)` → フィールド名 "title" で記録
- **tags**: `plaintext.tags.iter().any(|t| regex.is_match(t))` → フィールド名 "tags" で記録
- **テンプレート本文**: record_type が TEMPLATE の場合のみ、本文をマッチ対象とする

### 6. DiaryCore::search() メソッド (core/src/lib.rs)

**信頼性**: 🔵 *設計文書 architecture.md: Facade パターン*

```rust
impl DiaryCore {
    pub fn search(&self, query: &SearchQuery) -> Result<SearchResults, DiaryError> {
        search::search_entries(&self.vault_path, &self.engine, query)
    }
}
```

**TC-B01-01**: body に "今日は天気が良い" を含むエントリを作成し、パターン "天気" で検索して当該エントリが `SearchResults.matches` に含まれ、`matched_field` が "body" であることを検証する。

**TC-B01-02**: title が "2026年4月の振り返り" のエントリを作成し、パターン "振り返り" で検索して当該エントリが返却され、`matched_field` が "title" であることを検証する。

**TC-B01-03**: tags に "日記/旅行" を持つエントリを作成し、パターン "旅行" で検索して当該エントリが返却され、`matched_field` が "tags" であることを検証する。

**TC-B01-04**: テンプレートレコードの本文に "## 日報テンプレート" を含むデータを作成し、パターン "日報" で検索してテンプレートがマッチすることを検証する。

**TC-B01-05**: vault にエントリを3件作成し、どのフィールドにもマッチしないパターン "ZZZZNOTFOUND" で検索して `matched_entry_count` が 0、`matches` が空であることを検証する。

**TC-B01-06**: body に "会議は2026-04-08に開催" を含むエントリを作成し、パターン `\d{4}-\d{2}-\d{2}` で検索してマッチすることを検証する。regex 構文が正しく解釈されることの確認。

**TC-B05-01**: 10行のbodyを持つエントリを作成し、5行目にマッチする検索を実行。`context_lines` デフォルト (2) で `ContextBlock.lines` が3行目〜7行目（計5行）を含むことを検証する。

**TC-B05-02**: 同上のエントリに対して `context_lines = 5` で検索し、`ContextBlock.lines` が1行目〜10行目（全行）を含むことを検証する。

**TC-B08-01**: `EntryPlaintext` に `ZeroizeOnDrop` が derive されていることを型レベルで確認する。検索完了後に中間の復号データが保持されていないことを、`SearchResults` 内に平文 body が格納されていないことで間接的に検証する。

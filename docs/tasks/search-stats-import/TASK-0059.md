## 実装詳細

### 1. Search サブコマンド定義 (cli/src/main.rs)

**信頼性**: 🔵 *REQ-B01: pq-diary search <PATTERN>*

clap derive で Search サブコマンドを追加:

```rust
/// Search entries by regex pattern.
#[derive(Args)]
struct SearchArgs {
    /// Regex pattern to search for.
    pattern: String,

    /// Filter by tag (prefix match).
    #[arg(long)]
    tag: Option<String>,

    /// Number of context lines around matches (default: 2).
    #[arg(long, default_value = "2")]
    context: usize,

    /// Show match count only.
    #[arg(long)]
    count: bool,
}
```

`Command` enum に `Search(SearchArgs)` バリアントを追加する。

### 2. cmd_search() コマンドハンドラ (cli/src/commands.rs)

**信頼性**: 🔵 *REQ-B04〜B07 + 設計文書 VaultGuard パターン*

```rust
pub fn cmd_search(cli: &Cli, args: &SearchArgs) -> anyhow::Result<()>
```

処理フロー:
1. `get_password()` でパスワード取得
2. `DiaryCore::new()` + `unlock()`
3. `VaultGuard` 取得（Drop で `lock()` が保証される）
4. `SearchQuery` を構築:
   - `pattern`: `args.pattern.clone()`
   - `tag_filter`: `args.tag.clone()`
   - `context_lines`: `args.context`
   - `count_only`: `args.count`
5. `core.search(&query)?` を呼び出し
6. 結果表示（下記フォーマット）
7. `VaultGuard` drop で自動 lock

### 3. 出力フォーマット

**信頼性**: 🟡 *REQ-B04: grep 慣例から推測*

通常モード:
```
{id_prefix} {date} {title}
  {line_num}: {context_line}
  {line_num}: {context_line}
> {line_num}: {match_line}
  {line_num}: {context_line}
  {line_num}: {context_line}
--
```

各 `SearchMatch` について:
- 1行目: UUID の先頭8文字 + 日付 (YYYY-MM-DD) + タイトル
- 2行目以降: `ContextBlock` の各行を表示
  - マッチ行は `> ` プレフィックス付き
  - 非マッチ行は `  ` プレフィックス付き
- エントリ間は `--` で区切り

### 4. --count モード

**信頼性**: 🟡 *REQ-B07: grep -c 慣例*

`args.count` が `true` の場合、マッチ結果の詳細を表示せず、マッチエントリ数のみを表示:

```
{matched_entry_count} entries matched
```

### 5. --tag フィルタ

**信頼性**: 🟡 *REQ-B06: 既存 list コマンドとの一貫性*

`SearchQuery.tag_filter` に値をセットすることで、コアモジュール側で `Tag::is_prefix_of()` による前方一致フィルタが適用される。CLI 側で追加のフィルタ処理は不要。

### 6. 無効パターンのエラーハンドリング

**信頼性**: 🟡 *EDGE-001: 無効正規表現の明確なエラーメッセージ*

`core.search()` が `DiaryError::Search` を返した場合、ユーザーフレンドリーなエラーメッセージを表示:

```
Error: Invalid regex pattern: <regex error detail>
```

### テスト詳細

**TC-B04-01**: vault にエントリを作成し、`cmd_search()` を呼び出して stdout キャプチャし、出力が `{8桁hex} {YYYY-MM-DD} {title}` 形式であることを正規表現で検証する。

**TC-B05-03**: `context = 0` で検索を実行し、マッチ行のみが出力され、前後のコンテキスト行が含まれないことを検証する。

**TC-B06-01**: タグ "日記" と "技術" のエントリを作成し、`--tag "日記"` で検索してタグ "日記" のエントリのみが結果に含まれることを検証する。

**TC-B07-01**: 3件中2件にマッチするパターンで `--count` 検索を実行し、出力が "2 entries matched" であることを検証する。

**TC-B07-02**: マッチなしパターンで `--count` 検索を実行し、出力が "0 entries matched" であることを検証する。

**TC-EDGE-01**: 無効な正規表現 `[invalid` を渡して `cmd_search()` を呼び出し、終了コードが非ゼロで stderr にエラーメッセージが含まれることを検証する。

**TC-EDGE-02**: エントリ0件の vault に対して検索を実行し、"No matches found" が表示されることを検証する。

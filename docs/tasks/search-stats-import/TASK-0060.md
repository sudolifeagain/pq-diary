## タスク概要

vault 内の全エントリを走査して統計情報を集計するコアモジュールを実装する。エントリ総数、タグ分布、文字数統計、日別活動データなどを収集し、`VaultStats` 構造体として返却する。全統計フィールドは `Serialize` を derive し、JSON 出力に対応する。

## 実装詳細

### 1. stats モジュール新規作成 (core/src/stats.rs)

**信頼性**: 🔵 *REQ-C01, REQ-C02 + types.rs 参照*

以下の型を定義する（全て `Serialize` derive）:

- **`VaultStats`**: vault 全体の統計データ
  - `entry_count: usize` — エントリ総数
  - `tag_count: usize` — ユニークタグ数
  - `first_entry_date: Option<u64>` — 最初のエントリ日付 (Unix timestamp)
  - `last_entry_date: Option<u64>` — 最後のエントリ日付 (Unix timestamp)
  - `active_days_30d: usize` — 直近30日のアクティブ日数
  - `char_stats: CharStats` — 文字数統計
  - `tag_distribution: Vec<TagCount>` — タグ分布（上位10タグ、降順）
  - `daily_activity: Vec<DailyActivity>` — 日別活動データ

- **`CharStats`**: 文字数統計
  - `total: usize` — 全エントリの合計文字数
  - `average: usize` — 平均文字数
  - `max: usize` — 最大文字数

- **`TagCount`**: タグ名と出現回数
  - `tag: String` — タグ名
  - `count: usize` — 出現エントリ数

- **`DailyActivity`**: 日別活動
  - `date: String` — "YYYY-MM-DD" 形式
  - `count: usize` — 当日のエントリ数

### 2. collect_stats() 関数

**信頼性**: 🔵 *REQ-C02, REQ-C06: 全エントリ走査で本文復号が必要*

```rust
pub fn collect_stats(
    vault_path: &Path,
    engine: &CryptoEngine,
) -> Result<VaultStats, DiaryError>
```

処理フロー:
1. `read_vault()` で vault ヘッダ + レコード一覧取得
2. 集計用の中間変数を初期化:
   - `entry_count = 0`
   - `total_chars = 0`, `max_chars = 0`
   - `tag_counts: HashMap<String, usize>`
   - `daily_counts: HashMap<String, usize>`
   - `first_date: Option<u64> = None`, `last_date: Option<u64> = None`
3. 各レコード (record_type == ENTRY) について:
   a. `engine.decrypt(iv, ciphertext)` → `Zeroizing<Vec<u8>>`
   b. `serde_json::from_slice()` → `EntryPlaintext`
   c. 集計:
      - `entry_count += 1`
      - `total_chars += body.chars().count()`
      - `max_chars = max(max_chars, body.chars().count())`
      - `daily_counts[date_string] += 1`
      - 各タグについて `tag_counts[tag] += 1`
      - `first_date = min(first_date, created_at)`
      - `last_date = max(last_date, created_at)`
   d. `EntryPlaintext` は `ZeroizeOnDrop` で自動 zeroize
4. `VaultStats` を構築:
   - `avg_chars = if entry_count > 0 { total_chars / entry_count } else { 0 }`
   - `tag_distribution`: `tag_counts` を count 降順ソート → 上位10件
   - `tag_count`: ユニークタグの総数
   - `active_days_30d`: `daily_counts` から直近30日間のキーを数える
   - `daily_activity`: `daily_counts` を日付昇順ソートして `DailyActivity` に変換

## テスト要件

| テストID | テスト内容 | 信頼性 |
|----------|-----------|--------|
| TC-C01-01 | エントリ総数が正しい | 🔵 |
| TC-C01-02 | 最初/最後の日付が正しい | 🔵 |
| TC-C01-03 | 文字数統計（合計/平均/最大）が正しい | 🟡 |

| TC-C01-05 | 空vaultでエラーにならず | 🟡 |

### テスト詳細

**TC-C01-01**: vault に3件のエントリを作成し、`collect_stats()` を呼び出して `entry_count` が 3 であることを検証する。

**TC-C01-02**: 異なる日付のエントリを3件作成（2026-01-01, 2026-04-01, 2026-04-08）し、`first_entry_date` が 2026-01-01 の timestamp、`last_entry_date` が 2026-04-08 の timestamp であることを検証する。

**TC-C01-03**: body の文字数がそれぞれ 100, 200, 300 文字のエントリを3件作成し、`char_stats.total` が 600、`char_stats.average` が 200、`char_stats.max` が 300 であることを検証する。

**TC-C01-04**: タグ "日記" を5件、"技術" を3件、"旅行" を1件、他7種を各1件のエントリを作成し、`tag_distribution` の先頭が "日記" (count=5)、2番目が "技術" (count=3) であり、全体が降順ソートされており、最大10件であることを検証する。

**TC-C01-05**: エントリ0件の vault に対して `collect_stats()` を呼び出し、エラーにならず `entry_count` が 0、`first_entry_date` が `None`、`char_stats.total` が 0 であることを検証する。
